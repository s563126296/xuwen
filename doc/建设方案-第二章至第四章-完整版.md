# 第二章 现状及需求分析

## 2.1 现状分析

### 2.1.1 徐闻县交通区位概况

徐闻县位于中国大陆最南端，是琼州海峡北岸的重要交通枢纽，承担着粤琼两省陆路交通的咽喉功能。徐闻港（新港）作为琼州海峡主要客货滚装运输港口，是进出海南岛的核心通道之一。2023年第一季度，徐闻港车辆吞吐量约157万辆次，同比增长52%，旅客吞吐量约529万人次，同比增长68%，充分体现了琼州海峡通道的交通压力持续增长态势。

**交通基本数据：**

- 日均进港车辆：约8,000辆（平日），春运高峰期进港方向日均车流量可达50,000辆次
- 港口日均发船班次：约100+班次（徐闻港约86班/日 + 海安新港约21班/日）
- 单船载车量：35-120辆（主力船型60-80辆，大型船可达120辆以上）
- 港口日消化能力：约8,000-10,000辆（受泊位、航次、单船载车量限制）
- 主要进港通道：进港大道（双车道至三车道，约14.8公里）、徐海路（三车道，约4.2公里）、环半岛公路（三车道，约7.3公里），三条道路合计约20公里以上
- 拥堵高峰时段：中午至晚上八九点
- 历史最严重拥堵：2023年春运期间，进港大道、徐海大道、环半岛公路三条主要道路全线拥堵，排队长度约20公里，路上动态滞留车辆约1万辆，持续约5天
- 琼州海峡年停航次数：每年因台风、大雾等恶劣天气停航约10-20次（台风季7-10月尤为频繁），每次停航后恢复通航极易引发集中出行和大规模拥堵

由于港口运力有限（日消化能力约8,000-10,000辆），当车辆到达速率超过港口消化能力时，极易在进港通道形成大规模拥堵。春运期间日均车流量达平日的6倍以上，远超港口消化能力，拥堵问题尤为突出。此外，琼州海峡每年因台风、大雾等恶劣天气停航约10-20次，停航期间车辆积压，恢复通航后集中出行形成"堰塞湖效应"，是除春运外最主要的拥堵诱因。

### 2.1.2 现有交通管理手段

目前徐闻县针对进港交通拥堵已建立了一套以"预约管控"为核心的管理体系：

**（1）预约制管控**

所有车辆进入徐闻港必须提前通过预约系统预约，未预约车辆不允许进入港口区域。这是控制进港车辆总量的核心手段。

**（2）道路分流与临时停车**

在拥堵发生或逐步加剧时，徐闻县政府动态调整道路功能：
- 将次要道路临时划为停车区域，降低主要道路车辆通行压力；
- 保障非进港车辆的正常通行；
- 保障进港出租车（旅客坐船）的优先通行。

**（3）人工设卡检查**

在即将拥堵或拥堵逐步增加时，在关键路口设置检查点，安排人员核验车辆预约状态，未预约车辆不允许继续前行。

**（4）信息发布**

通过道路诱导屏、广播等方式发布分流指引和停车区指引信息。

**（5）视频监控**

主要路段和港口区域已部署视频监控摄像头（海康/大华），用于日常巡查和事件发现。

### 2.1.3 现有信息化基础

| 系统/设施 | 现状 | 备注 |
|-----------|------|------|
| 预约系统 | 已运行 | 控制进港车辆总量 |

## 2.2 存在问题

尽管徐闻县已建立了以预约制为核心的交通管控体系，但在实际运行中仍存在以下突出问题：

### 2.2.1 拥堵预判能力不足

当前交通管理以"被动响应"为主——等到拥堵已经发生或明显加剧后才启动应对措施。缺乏基于预约数据、实时路况、气象信息等多源数据的综合预判能力，无法在拥堵形成之前提前介入。

**具体表现：**
- 无法根据未来时段的预约量预判即将到来的车流高峰；
- 无法结合气象预报（如台风恢复通航后的集中出行）提前预警；
- 从发现拥堵到启动分流措施存在明显时间差。

### 2.2.2 多源数据未打通

预约数据、实时路况数据、气象数据、视频监控数据分散在不同系统中，缺乏统一的数据汇聚和关联分析平台。管理人员需要在多个系统间切换查看，无法形成对交通态势的全局认知。

**具体表现：**
- 预约系统只能看到预约量，看不到实时路况；
- 路况数据只反映当前状态，无法结合预约数据预判趋势；
- 气象预警与交通管理决策之间缺乏自动化联动。

### 2.2.3 决策依赖人工经验

何时启动道路分流、何时设置检查点、向哪些预约用户发送通知、建议调整到什么时间到达——这些关键决策目前完全依赖管理人员的个人经验判断，缺乏数据支撑和标准化流程。

**具体表现：**
- 不同值班人员对同一态势的判断和响应可能不同；
- 缺乏量化的分级响应标准（如拥堵指数达到多少启动分流）；
- 预约用户通知的时机和内容缺乏精准性。

### 2.2.4 缺少空中视角能力

当拥堵发生时，地面视角受限，难以快速获取拥堵的全貌信息（拥堵长度、滞留车辆数量、拥堵蔓延方向等）。目前尚未配置无人机，无法提供空中巡查、喊话疏导等辅助手段。

### 2.2.5 缺乏复盘与持续优化能力

每次拥堵处置结束后，缺乏系统化的数据记录和效果评估机制。历史处置经验未能沉淀为可量化、可对比的数据资产，无法支撑策略的持续优化。

**具体表现：**
- 无法回答"上次春运分流措施效果如何"；
- 无法对比不同处置策略的缓解时间和效果；
- 领导汇报材料需要手动整理，缺乏自动化报告能力。

- AI预测技术可实现基于多源数据的拥堵预判；

- 无人机技术可提供低成本的空中巡查和应急服务能力；
- 数据融合技术可打通多个信息孤岛，形成统一决策视图。

# 第三章 总体设计

## 3.1 设计原则

### 3.1.1 预判优先

从传统的"拥堵发生后被动响应"转变为"基于数据提前预判、主动干预"。通过预约数据、路况趋势、气象信息的综合分析，在拥堵形成之前启动预防措施，将问题消灭在萌芽阶段。

### 3.1.2 AI辅助、人工决策

AI负责数据分析、态势研判、策略推荐，人负责最终决策确认和执行指挥。系统定位为"智能参谋"而非"自动驾驶"，确保关键决策始终有人在回路中。

### 3.1.3 平战结合

系统在日常状态下提供态势监控和数据分析能力，在拥堵或应急状态下无缝切换为指挥调度模式。同一套系统、同一套数据，覆盖从日常到应急的全场景。

### 3.1.4 数据驱动

以预约系统数据为核心决策依据，融合路况、气象、视频等多源数据，用数据说话、用数据决策、用数据评估，替代经验驱动的管理模式。

### 3.1.5 务实可行

基于现有基础设施条件进行设计，不依赖尚不具备的能力（如信号灯远程控制）。策略执行通过诱导屏信息发布、人工设卡、无人机辅助等现有或即将配置的手段落地。

## 3.2 设计思路

### 3.2.1 核心理念

本系统以"**10秒判断态势、1分钟拿到方案、全程看到AI在干活**"为核心设计理念：

- **10秒判断态势**：进入系统即可通过AI摘要栏、风险等级色标、关键指标卡片，在10秒内掌握当前交通全局状态；
- **1分钟拿到方案**：AI根据当前拥堵阶段自动推荐对应处置策略，指挥员1分钟内即可获得可执行的方案；
- **全程看到AI在干活**：从预判预警、策略推荐、执行监控到效果评估，AI贯穿处置全流程，用户始终能看到AI的分析过程和结论。

### 3.2.2 业务模式设计

系统设置三大业务模式，覆盖交通管理全场景：

| 模式 | 定位 | 核心场景 |
|------|------|---------|
| 总览监控模式 | 日常值班，一屏看全局 | 实时态势感知、AI预警、风险评估 |
| 指挥处置模式 | 拥堵/应急响应，策略执行 | 分级响应、策略推荐、指令下达、执行监控 |
| 统计分析模式 | 历史复盘，报告生成 | 趋势分析、策略评估、报告导出 |

三大模式之间的数据流转关系：
- 总览模式实时数据 → 统计分析模式历史库；
- 指挥模式策略执行记录 → 统计分析模式效果评估；
- 统计分析模式优化建议 → 指挥模式策略库更新。

### 3.2.3 分阶段递进响应机制

基于徐闻县现有的拥堵处置实践，系统设计四级递进响应机制：

**第一阶段：常态管控（绿色）**
- 系统状态：日常监控，预约数据正常
- 系统动作：实时展示路况和预约量，AI持续评估风险
- 人工动作：无需干预

**第二阶段：预警通知（黄色）**
- 触发条件：AI预判未来30分钟内拥堵概率 > 60%
- 系统动作：弹窗预警 + AI建议向预约用户发送通知（调整到达时间）
- 人工动作：确认是否发送通知，准备分流预案
- 无人机：可起飞确认实际路况，为决策提供空中视角依据

**第三阶段：道路分流（橙色）**
- 触发条件：拥堵指数持续上升，主要道路车速明显下降
- 系统动作：AI推荐分流方案（哪些道路划为停车区、分流路线建议）+ 诱导屏信息推送
- 人工动作：确认分流方案，安排人员设卡检查预约状态
- 无人机：监控分流执行效果，统计拥堵车辆数量和距离，喊话引导车辆进入停车区

**第四阶段：全面管控（红色）**
- 触发条件：完全拥堵，主要道路车辆无法移动
- 系统动作：资源调度建议 + 全员通知
- 人工动作：全面管控，持续分流+停车区管理
- 无人机：全路段巡查，统计拥堵长度和滞留车辆数量，大范围喊话疏导，数据实时回传大屏

### 3.4.2 业务架构

**核心业务流程：**

系统的核心业务流程为"感知→研判→决策→执行→反馈"五步闭环：

1. **数据感知**：实时采集预约数据、路况数据、气象数据、视频数据；
2. **AI研判**：融合多源数据，计算风险等级，识别当前拥堵阶段；
3. **策略推荐**：根据当前阶段，AI推荐对应的处置策略和具体动作；
4. **执行落地**：指挥员确认后，通过通讯系统下达指令，一线人员执行；
5. **效果反馈**：实时监控执行效果，偏差超阈值时AI建议调整。

**角色协作：**

| 角色 | 在系统中的职责 |
|------|--------------|
| AI系统 | 数据分析、风险预判、策略推荐、偏差检测、效果评估 |
| 指挥员 | 查看AI分析结论、确认执行策略、下达指令、处理偏差 |
| 一线人员 | 接收任务（设卡/分流/无人机操作）、反馈执行进度 |
| 外部系统 | 提供数据（预约/路况/气象）、接收指令（诱导屏/通知推送） |

### 3.4.3 AI引擎设计

AI引擎是本系统的核心能力，由五个子引擎组成：

**（1）风险预测引擎**

输入：预约数据（未来时段预约量）+ 实时路况（拥堵指数、车速）+ 气象信息（台风/大雾预警）

输出：未来30分钟拥堵概率、风险等级（绿/黄/橙/红）

计算逻辑：
- 预约饱和度 = 未来2小时预约量 / 港口小时消化能力
- 路况恶化趋势 = 近15分钟拥堵指数变化率
- 气象影响系数 = 根据天气类型赋权（台风恢复通航=高，大雾=中，正常=低）
- 综合风险分 = 加权计算 → 映射到四级预警

**（2）阶段识别引擎**

基于实时指标自动判断当前处于四个阶段中的哪一个：

| 阶段 | 判断条件 |
|------|---------|
| 常态（绿色） | 拥堵指数 < 4，预约饱和度 < 70% |
| 预警（黄色） | 拥堵指数 4-6 或 预约饱和度 70%-90% |
| 分流（橙色） | 拥堵指数 6-8，主要道路车速 < 20km/h |
| 全面管控（红色） | 拥堵指数 > 8，主要道路车速 < 5km/h |

**（3）策略推荐引擎**

根据当前阶段，从策略库中匹配对应的处置方案：

| 阶段 | 推荐策略 |
|------|---------|
| 预警 | 向预约用户推送通知，建议调整到达时间 |
| 分流 | 推荐分流路线 + 停车区划定方案 + 诱导屏信息内容 |
| 全面管控 | 无人机巡查路线 + 设卡点位 + 资源调度方案 |

**（4）偏差监控引擎**

策略执行过程中，持续监控实际效果与预期的偏差：
- 每30秒采集一次实时指标；
- 与策略预期效果对比，偏差超过±15%时触发预警；
- 自动生成调整建议（如：分流效果不佳，建议增加停车区）。

**（5）效果评估引擎**

策略执行完成后，自动评估处置效果：
- 计算缓解时间（从执行到恢复正常的时长）；
- 计算缓解率（执行前后拥堵指数变化）；
- 生成效果评分，归档到历史数据库；
- 为统计分析模式提供数据支撑。

# 第四章 整合共享与业务协同

## 4.1 系统整合

### 4.1.1 整合目标

本项目不是孤立建设新系统，而是在现有信息化基础上进行整合升级。通过统一的数据接入平台，将分散在各部门、各系统中的交通相关数据汇聚到智慧交通指挥系统中，形成统一的决策视图。

### 4.1.2 与预约系统对接（核心）

预约系统是本项目最核心的数据来源，其数据直接决定了拥堵预判的准确性。

| 对接内容 | 说明 | 优先级 |
|---------|------|--------|
| 实时预约量查询 | 获取当前及未来各时段的预约车辆数量 | P0 |
| 预约时段分布 | 按小时统计预约量，识别高峰时段 | P0 |
| 预约车辆类型 | 区分小客车、货车、大巴等类型 | P1 |
| 用户通知推送 | 向预约用户发送到达时间调整建议 | P0 |
| 预约状态变更 | 获取取消、改签等变更信息 | P1 |

**对接方式：** REST API（实时查询）+ WebSocket（变更推送）

**关键能力：** 系统需要具备向预约用户推送通知的能力，这是"预警通知"阶段的核心执行手段。在即将拥堵时，系统通过预约平台向即将到达的用户发送消息，建议其延后出发或调整到达时间，从源头削减车流高峰。

### 4.1.3 与高德地图对接

高德地图提供实时路况数据，是拥堵态势感知的基础数据源。

| 对接内容 | 说明 | 优先级 |
|---------|------|--------|
| 实时拥堵指数 | 主要道路的拥堵等级和指数 | P0 |
| 实时车速 | 各路段平均车速 | P0 |
| 拥堵长度 | 拥堵路段的排队长度 | P0 |
| 路况趋势 | 近15分钟的路况变化趋势 | P1 |

**对接方式：** REST API，1分钟轮询

**坐标系：** GCJ-02（高德坐标系），系统全局统一使用此坐标系。

### 4.1.4 与气象系统对接

气象数据是拥堵预判的重要辅助因素，特别是台风停航后恢复通航、大雾等场景。

| 对接内容 | 说明 | 优先级 |
|---------|------|--------|
| 气象预警信息 | 台风、大雾、暴雨等预警 | P0 |
| 风力/能见度 | 实时风力等级和能见度数据 | P1 |
| 未来天气预报 | 未来24-72小时天气趋势 | P1 |
| 停航/复航信息 | 海事部门发布的停航和恢复通航通知 | P0 |

**对接方式：** REST API，15分钟轮询（预警信息实时推送）

**关键场景：** 台风停航后恢复通航是最容易引发大规模拥堵的场景。系统需要在收到复航通知后，立即结合预约数据评估即将到来的车流高峰，提前启动预警。

### 4.1.5 与视频监控系统对接

利用现有视频监控设施，为指挥决策提供实时画面支撑。

| 对接内容 | 说明 | 优先级 |
|---------|------|--------|
| 实时视频流 | 主要路段和港口区域摄像头画面 | P0 |
| 视频回放 | 历史视频调取（复盘使用） | P2 |

**对接方式：** RTSP/HLS 视频流协议

### 4.1.6 与无人机系统对接

未来配置的2台无人机将作为拥堵处置各阶段的重要辅助手段。

| 对接内容 | 说明 | 优先级 |
|---------|------|--------|
| 视频回传 | 无人机实时画面接入大屏 | P0 |
| 任务状态 | 无人机起飞/巡查/返航状态 | P1 |
| 飞行轨迹 | 在地图上展示无人机实时位置 | P1 |

**对接方式：** RTSP视频流 + 无人机管理平台API

**使用场景（贯穿多个阶段）：**
- 预警阶段：起飞确认实际路况，为决策提供空中视角依据；
- 分流阶段：监控分流执行效果，统计拥堵车辆数量和距离，喊话引导车辆进入停车区；
- 全面管控阶段：全路段巡查，统计拥堵长度和滞留车辆数量，大范围喊话疏导。

### 4.1.7 与诱导屏/广播系统对接

诱导屏是策略执行的重要信息发布渠道。

| 对接内容 | 说明 | 优先级 |
|---------|------|--------|
| 信息发布 | 向诱导屏推送分流指引、停车区指引 | P0 |
| 状态查询 | 查询诱导屏当前显示内容和在线状态 | P2 |

**对接方式：** MQTT 消息推送

### 4.2.2 数据标准规范

为确保多源数据的有效融合，制定以下数据标准：

**（1）坐标系统一**
- 全系统统一使用 GCJ-02 坐标系（高德坐标系）；
- 外部数据如使用 WGS-84 坐标系，接入时自动转换。

**（2）时间格式统一**
- 统一使用 ISO 8601 格式：`YYYY-MM-DDTHH:mm:ssZ`；
- 时区统一为东八区（UTC+8）。

**（3）接口规范统一**
- RESTful API 遵循 OpenAPI 3.0 规范；
- 请求/响应统一使用 JSON 格式；
- 统一错误码体系（业务错误码 + HTTP状态码）。

**（4）数据质量要求**
- 数据完整性：关键字段不允许为空；
- 数据时效性：超过有效期的数据标记为过期；
- 数据异常检测：接入层自动检测数据断流和异常值。

### 4.2.3 数据安全与权限

| 安全措施 | 说明 |
|---------|------|
| 传输加密 | 所有API调用使用HTTPS |
| 接口认证 | Token认证机制，定期轮换 |
| 访问控制 | 按角色分配数据访问权限 |
| 数据脱敏 | 预约用户个人信息在系统中脱敏展示 |

## 4.3 业务协同

### 4.3.1 跨部门协同机制

智慧交通指挥系统涉及多个部门的协同配合：

| 部门 | 协同职责 | 系统支撑 |
|------|---------|---------|
| 交通局 | 道路分流决策、诱导屏信息发布 | 分流方案推荐、诱导屏内容推送 |
| 公安局/交警 | 设卡检查、交通管制 | 设卡点位建议、预约核验信息 |
| 应急管理局 | 台风/极端天气应急联动 | 气象预警联动、应急预案触发 |
| 无人机飞手 | 无人机起飞、巡查、喊话 | 任务派发、巡查路线建议 |
| 预约平台运营方 | 用户通知推送 | 通知内容生成、推送指令 |

### 4.3.2 分阶段协同流程

**阶段一：常态管控**
- 协同方：无（系统自动监控）
- 系统动作：持续采集数据，AI评估风险

**阶段二：预警通知**
- 协同方：预约平台运营方 + 无人机飞手（视情况）
- 系统动作：生成通知内容 → 推送至预约平台 → 平台向用户发送
- 无人机动作：可起飞确认实际路况，为是否启动分流提供空中视角依据
- 协同要点：通知内容需包含建议到达时间、当前路况、替代方案

**阶段三：道路分流**
- 协同方：交通局 + 公安局/交警 + 无人机飞手
- 系统动作：推荐分流方案 → 诱导屏信息推送 → 设卡点位建议
- 无人机动作：监控分流执行效果，统计拥堵车辆数量和距离，喊话引导车辆进入停车区
- 协同要点：
  - 交通局确认分流路线和停车区划定；
  - 公安局安排人员到检查点核验预约；
  - 确保非进港车辆和出租车通行不受影响；
  - 无人机统计数据实时回传大屏，辅助指挥员评估分流效果。

**阶段四：全面管控**
- 协同方：全部门 + 无人机飞手
- 系统动作：全员通知 → 资源调度
- 无人机动作：全路段巡查，统计拥堵长度和滞留车辆数量，大范围喊话疏导
- 协同要点：
  - 无人机飞手按系统建议路线起飞巡查；
  - 无人机统计数据（拥堵长度、车辆数）回传系统；
  - 喊话内容由指挥员通过系统确认后执行。

### 4.3.3 通讯协同机制

系统内置通讯能力，支撑跨部门实时协同：

| 通讯方式 | 适用场景 |
|---------|---------|
| 系统消息 | 任务派发、进度反馈、状态通知 |
| 快捷指令 | 预设常用指令（如"启动分流"、"设卡检查"），一键下达 |
| 视频连线 | 与现场人员/无人机画面实时连线 |

### 4.4.2 核心接口规范说明

**接口1：预约数据查询**

- 接口方式：GET请求
- 请求参数：查询时间范围（如未来2小时）、统计粒度（按小时/30分钟/15分钟）
- 返回内容：预约总量、各时段预约分布（含车辆类型细分）、预约饱和度

**接口3：用户通知推送**

- 接口方式：POST请求
- 请求参数：目标时段范围、通知消息内容、建议到达时间、优先级
- 返回内容：已通知用户数量、推送成功率

**接口10：诱导屏信息发布**

- 接口方式：MQTT消息推送
- 消息内容：显示文字内容、信息类型（分流/停车/预警）、优先级、显示时长
