# 04 应急模式（Emergency Mode）— 产品需求文档

> 版本：V1.0 | 日期：2026-04-13
> 优先级：P0 | 状态：需求定义中

---

## 一、模式定位

台风停航、极端天气等应急场景下的**指挥调度中枢**。当港口停运、车辆大规模滞留时，为县指挥部提供**滞留预测 + 资源调度 + 跨部门协同 + 特殊车辆追踪**的完整应急指挥能力。

核心价值：从"台风来了临时开会"升级为"系统提前预测滞留规模，自动生成应急预案，实时调度资源"。

**与指挥模式的边界：**

| 场景 | 使用模式 | 判断依据 |
| ---- | -------- | -------- |
| 日常拥堵（拥堵指数 > 4） | 指挥模式 | 拥堵是临时的，策略执行后可缓解 |
| 重度拥堵但无停航（拥堵指数 > 6） | 指挥模式 | 港口仍在运营，运力可消化积压 |
| 台风预警 + 港口仍运营 | 指挥模式 + 顶部预警横幅 | 港口未停，但需提前准备 |
| 港口停航（不论原因） | **应急模式** | 运力归零，车辆只进不出，性质变了 |
| 台风预警 + 港口已停航 | **应急模式** | 最严重场景 |
| 停航结束 + 大量滞留车辆待消化 | **应急模式**（恢复阶段） | 虽然复航但积压未消，仍需应急调度 |

## 二、触发与退出规则

### 触发条件

| 条件 | 行为 | 说明 |
| ---- | ---- | ---- |
| 港口停航通知（P-14） | **自动切换**应急模式 + 全屏红色横幅 | 港口方会提前通知县政府，比公众更早 |
| 气象台发布台风橙色/红色预警 | **自动切换**应急模式 + 全屏红色横幅 | 橙色以上预警意味着停航概率极高 |
| 气象台发布台风蓝色/黄色预警 | 顶部黄色横幅提示，建议手动切换 | 蓝色/黄色预警不一定停航 |
| 手动切换 | 任何时候均可从导航栏进入 | 用于演练或提前准备 |

### 退出条件

```text
应急模式退出需要同时满足以下条件：
  1. 港口已复航 且 复航时间 > 2小时
  2. 滞留车辆数 < 200辆（积压已基本消化）
  3. 进港大道拥堵指数 < 4（道路恢复正常）

满足后：
  系统提示"应急状态已解除，是否切换回总览模式？"
  需二次确认（防止误操作）

特殊情况：
  如果复航但滞留仍严重（> 500辆）→ 保持应急模式
  如果滞留已消化但尚未正式复航 → 保持应急模式
```

## 四、功能清单

### A. 滞留预测面板（左栏上部）

应急模式最核心的能力——精确预测滞留规模，为资源调度提供依据。

| 编号 | 功能项 | 说明 | 数据刷新 |
| ---- | ------ | ---- | -------- |
| A-01 | 预计停航时长 | 基于气象预报+历史台风数据推算 | 30分钟 |
| A-02 | 当前滞留车辆数 | 已到达但无法上船的车辆实时数 | 5分钟 |
| A-03 | 预计滞留峰值 | 停航期间最大滞留车辆数预测 | 15分钟 |
| A-04 | 滞留车辆增速 | 每小时新增滞留车辆数 | 5分钟 |
| A-05 | 预计复航时间 | 基于气象预报推算的最早复航时间 | 30分钟 |
| A-06 | 复航后消化时间 | 复航后需要多久消化全部积压 | 15分钟 |
| A-07 | 滞留阶段标识 | 当前处于哪个阶段（见下方） | 实时 |

**滞留阶段定义：**

应急事件不是一个静态状态，而是一个有明确阶段的过程。每个阶段的重点工作不同：

```text
阶段1：预警期（停航前）
  触发：台风预警发布 / 收到港口即将停航的提前通知
  持续：通常2-12小时（取决于台风路径和港口决策速度）
  重点：
    - 预测滞留规模
    - 提前部署临时停车区
    - 通知各部门进入待命状态
    - 通过诱导屏/导航平台提醒司机"港口即将停航，建议暂缓出发"
  大屏显示：黄色/橙色横幅，"预计X小时后停航"

阶段2：停航初期（停航后0-6小时）
  触发：港口正式停航
  重点：
    - 截流已在途车辆（已出发的无法拦回，需安排停放）
    - 引导车辆进入临时停车区
    - 冷链车优先安排有电源的停放区域
    - 启动物资发放准备
  大屏显示：红色横幅，滞留数实时上涨

阶段3：滞留高峰期（停航后6小时-复航前）
  触发：滞留车辆数超过预测峰值的50%
  重点：
    - 物资发放（盒饭、饮用水）
    - 冷链车持续监控（燃油/电力保障）
    - 秩序维护（违停清理、应急车道保障）
    - 司机情绪安抚（发布复航预测信息）
    - 如停航超过48小时，启动二级物资补给
  大屏显示：红色横幅，滞留数趋于稳定或缓慢增长

阶段4：复航准备期（复航前2-4小时）
  触发：气象条件改善 + 港口通知即将复航
  重点：
    - 制定放行方案（先放冷链车、再放小客车、最后普通货车）
    - 引导临时停车区车辆有序回到进港大道
    - 通知各部门准备撤收
    - 预测复航后道路压力（积压车辆集中涌入）
  大屏显示：橙色横幅，"预计X小时后复航"

阶段5：复航消化期（复航后）
  触发：港口恢复运营
  重点：
    - 分批放行滞留车辆（避免一次性涌入导致港口再次饱和）
    - 指挥模式策略同步启用（S-06限流+S-04信号灯+S-14交警）
    - 监控积压消化速率
    - 各部门逐步撤收
  大屏显示：黄色横幅，"港口已复航，积压消化中，剩余X辆"
  退出条件：滞留<200辆 + 拥堵指数<4 + 复航>2小时
```

**滞留预测算法：**

```text
核心公式：
  预计滞留峰值 = 停航期间持续到达车辆 + 停航时已在途车辆 + 停航时已在港排队车辆

分项计算：

1. 停航时已在港排队车辆（已知）
   = 当前待舶车辆数（港口闸机数据 P-01）

2. 停航时已在途车辆（推算）
   = 已购票但未到达港口的车辆
   数据源：港口售票系统（已售票 - 已进港）
   如售票数据不可用：用卡口数据推算
     近2小时进城方向流量均值 × 平均在途时间(约1小时)

3. 停航期间持续到达车辆（预测）
   分为两部分：
   a) 已购票车辆：售票系统中未来各时段已售票量（最精确）
   b) 未购票车辆（临时出发）：
      按历史同期同时段到达流量 × 衰减系数
      衰减系数 = f(预警发布时长, 诱导屏覆盖率)
      预警发布后：
        第1小时衰减10%（信息传播延迟）
        第2-4小时衰减30%（部分司机取消出行）
        第4小时后衰减50%（大部分人已知情）

示例计算：
  停航时已排队：500辆
  已在途：300辆（售票系统显示已购票未到达）
  预计停航48小时：
    已购票未来车辆：800辆（售票系统）
    未购票到达（衰减后）：48h × 15辆/h × 0.5 = 360辆
  
  预计滞留峰值 = 500 + 300 + 800 + 360 = 1,960辆
  
  冷链车占比（历史数据）：约8%
  预计冷链车 = 1,960 × 8% ≈ 157辆

复航后消化时间：
  港口正常运力：800辆/时
  复航初期运力（安全起见按70%）：560辆/时
  复航后持续到达流量：约200辆/时（积压期间被抑制的需求释放）
  净消化速率 = 560 - 200 = 360辆/时
  消化时间 = 1,960 ÷ 360 ≈ 5.4小时
```

### B. 特殊车辆追踪（左栏中部）

停航期间最紧急的问题是冷链车——不能熄火，长时间滞留会导致燃油耗尽、货物损失。

| 编号 | 功能项 | 说明 | 数据刷新 |
| ---- | ------ | ---- | -------- |
| B-01 | 冷链车实时数量 | 当前滞留区域内的冷链车数量 | 5分钟 |
| B-02 | 冷链车位置标记 | 在地图上用蓝色图标标记每辆冷链车位置 | 5分钟 |
| B-03 | 冷链车滞留时长 | 每辆冷链车已等待的时长，超过阈值告警 | 实时 |
| B-04 | 危化品车数量 | 危化品运输车实时数量 | 5分钟 |
| B-05 | 危化品车位置 | 地图红色图标标记，需与普通车辆保持安全距离 | 5分钟 |
| B-06 | 锂电池类车辆 | 应已被引导至海安新港，如仍在徐闻港区域则告警 | 事件触发 |
| B-07 | 特殊车辆汇总看板 | 冷链/危化品/锂电池分类统计+趋势 | 5分钟 |

**冷链车告警规则：**

```text
告警等级：
  黄色告警：冷链车滞留 > 6小时
    → 提示"建议安排加油/充电服务"
  
  橙色告警：冷链车滞留 > 12小时
    → 提示"货物损失风险增大，建议优先安排复航后首批上船"
    → 自动将该车辆加入"优先放行名单"
  
  红色告警：冷链车滞留 > 24小时
    → 提示"紧急：货物可能已受损，请现场确认车辆状态"
    → 通知现场交警/民政人员前往查看
    → 如车辆已熄火，启动紧急供电方案

冷链车燃油估算：
  冷链车制冷待机油耗：约3-5L/小时
  普通冷链车油箱：200-400L
  满油估计可待机时间：40-130小时
  但多数车辆出发时并非满油，按60%油量估算：24-78小时
  
  → 停航超过24小时时，必须排查冷链车燃油情况
  → 停航超过48小时时，需安排移动加油车服务
```

### C. 物资需求估算（左栏下部）

基于滞留预测自动计算应急物资需求，替代人工拍脑袋。

| 编号 | 功能项 | 说明 | 数据刷新 |
| ---- | ------ | ---- | -------- |
| C-01 | 滞留人数估算 | 滞留车辆数 × 平均每车人数 | 15分钟 |
| C-02 | 盒饭需求量 | 滞留人数 × 餐次 × 停航天数 | 15分钟 |
| C-03 | 饮用水需求量 | 滞留人数 × 日均用水量 × 停航天数 | 15分钟 |
| C-04 | 已发放物资量 | 各类物资已发放的累计数量 | 事件触发 |
| C-05 | 物资缺口预警 | 已备物资 vs 预计需求的差值 | 15分钟 |
| C-06 | 移动加油需求 | 冷链车燃油估算，是否需要调配加油车 | 1小时 |
| C-07 | 医疗需求估算 | 基于滞留人数和时长的基础医疗物资估算 | 1小时 |

**物资计算逻辑：**

```text
基础参数：
  每车平均人数：
    小型客车 2.5人
    货车/冷链车 1.5人
    加权平均（按车型分布）：约2.0人

盒饭需求：
  每人每天3餐
  需求量 = 滞留人数 × 3 × 预计停航天数 × 1.2（20%冗余）
  
  示例：滞留2000辆 → 约4000人
    停航2天：4000 × 3 × 2 × 1.2 = 28,800份盒饭
    停航3天：4000 × 3 × 3 × 1.2 = 43,200份盒饭

饮用水需求：
  每人每天2L（夏季3L）
  需求量 = 滞留人数 × 日均用水 × 预计停航天数 × 1.2
  
  示例（夏季台风）：4000人 × 3L × 2天 × 1.2 = 28,800L
    换算：约1,440箱（按20瓶/箱，500ml/瓶）

移动加油车需求：
  IF 停航预计 > 24小时 AND 冷链车 > 50辆：
    → 建议调配1辆移动加油车
  IF 停航预计 > 48小时 AND 冷链车 > 100辆：
    → 建议调配2辆移动加油车
  
  每辆移动加油车容量约5000L
  每辆冷链车加油约100L（补给用，非加满）
  1辆加油车可服务约50辆冷链车

物资缺口预警规则：
  已备物资 < 预计需求 × 50% → 🔴 严重不足，立即采购/调拨
  已备物资 < 预计需求 × 80% → 🟡 储备不足，建议补充
  已备物资 ≥ 预计需求 → 🟢 储备充足
```

### D. 应急资源部署地图（中央区域）

从日常GIS地图切换为应急部署地图，核心是展示"人在哪、物资在哪、车辆在哪"。

| 编号 | 功能项 | 说明 |
| ---- | ------ | ---- |
| D-01 | 高德深色底图 | 与指挥模式共用底图 |
| D-02 | 临时停车区标记 | 显示已启用的临时停车区位置、容量、当前使用率 |
| D-03 | 物资发放点标记 | 盒饭/饮用水发放点位置，标注当前库存量 |
| D-04 | 人员部署点位 | 交警、民政人员、志愿者的部署位置 |
| D-05 | 冷链车分布热力 | 冷链车聚集区域的热力图叠加 |
| D-06 | 滞留车辆密度 | 道路上滞留车辆的密度可视化（颜色深浅） |
| D-07 | 无人机巡查路线 | 无人机计划/实际飞行路线叠加 |
| D-08 | 加油车位置 | 移动加油车当前位置和服务状态 |
| D-09 | 应急通道标记 | 必须保持畅通的应急车道高亮显示 |
| D-10 | 点击交互 | 点击任意标记点 → 弹出详情（人员名单/库存/车辆信息） |

**临时停车区管理：**

```text
徐闻可用临时停车区（需提前勘察确认）：
  编号  位置            面积(㎡)   预估容量   距港口
  P-1   [待确认]        10,000     ~350辆    2km
  P-2   [待确认]        8,000      ~280辆    3km
  P-3   [待确认]        5,000      ~175辆    4km

停车区启用规则：
  滞留预测 < 500辆 → 仅启用P-1
  滞留预测 500-1000辆 → 启用P-1 + P-2
  滞留预测 > 1000辆 → 全部启用

停车区使用率监控：
  > 80% → 🟡 接近满载
  > 95% → 🔴 即将满载，启用下一个停车区
  
  冷链车专区：每个停车区预留10%面积给冷链车
  （靠近出口位置，便于优先放行）
```

### E. 视频/无人机画中画（中央区域）

**无人机调度规则：**

```text
徐闻现有无人机：政府自有几台大疆（需飞手服务）

巡查场景：
  1. 道路巡查：沿进港大道从城区到港口，查看排队长度和秩序
  2. 停车区巡查：巡视各临时停车区使用情况
  3. 事故/事件响应：接到报告后飞往指定位置查看

巡查频率：
  停航初期：每2小时巡查一次
  滞留高峰：每1小时巡查一次
  台风期间：视风力情况，风力>6级停飞

飞手服务需求：
  需要至少2名持证飞手（轮班）
  建议采购飞手外包服务（年费约10-15万）

    → E-01/E-03摄像头视频功能仍可用
```

## 五、应急响应等级体系

| 等级 | 触发条件 | 响应力度 | 参与部门 |
| ---- | -------- | -------- | -------- |
| Ⅳ级（蓝色） | 台风蓝色预警 / 停航预计<6小时 / 滞留<500辆 | 部门值班待命 | 公安交警+交通局 |
| Ⅲ级（黄色） | 台风黄色预警 / 停航6-24小时 / 滞留500-1500辆 | 启动应急预案，主要部门到位 | +民政+城管 |
| Ⅱ级（橙色） | 台风橙色预警 / 停航24-72小时 / 滞留1500-3000辆 | 全面应急，县领导坐镇指挥 | +应急管理局+港口方 |
| Ⅰ级（红色） | 台风红色预警 / 停航>72小时 / 滞留>3000辆 / 特大台风 | 最高等级，请求市级/省级支援 | 全部+上级支援 |

**等级自动判定逻辑：**

```text
等级 = max(天气等级, 停航时长等级, 滞留规模等级)

天气等级映射：
  蓝色预警 → Ⅳ级
  黄色预警 → Ⅲ级
  橙色预警 → Ⅱ级
  红色预警 → Ⅰ级

停航时长等级映射：
  <6小时 → Ⅳ级
  6-24小时 → Ⅲ级
  24-72小时 → Ⅱ级
  >72小时 → Ⅰ级

滞留规模等级映射：
  <500辆 → Ⅳ级
  500-1500辆 → Ⅲ级
  1500-3000辆 → Ⅱ级
  >3000辆 → Ⅰ级

取三者中最高等级。
例：黄色预警(Ⅲ) + 停航12小时(Ⅲ) + 滞留2000辆(Ⅱ) → 最终Ⅱ级

等级可手动上调（领导判断需要更高响应），不可手动下调至自动判定以下。
```

## 七、与指挥模式的衔接

### 7.1 从指挥模式切换到应急模式

```text
场景：正在指挥模式处理日常拥堵，突然收到港口停航通知

自动衔接逻辑：
  1. 保存指挥模式当前状态（正在执行的策略不中断）
  2. 自动切换到应急模式布局
  3. 指挥模式中正在执行的策略状态同步到应急模式：
     - S-02/S-03 分流 → 在应急地图上继续显示分流路线
     - S-04 信号灯配时 → 保持当前配时方案
     - S-14 交警部署 → 人员位置同步到应急地图
  4. 应急模式自动追加新任务：
     - 停止新车辆进入进港大道（升级S-06限流为全面截流）
     - 启用临时停车区
     - 通知民政启动物资准备
```

### 7.2 从应急模式切换到指挥模式（复航消化期）

```text
场景：港口复航，积压车辆需要消化

衔接逻辑：
  应急模式退出条件满足后 → 建议切换到指挥模式
  
  指挥模式自动继承：
  1. 积压车辆数 → 作为指挥模式的"影响车辆数"
  2. 分批放行方案 → 转化为指挥模式的S-06限流策略
  3. 交警部署 → 保持，直到积压完全消化
  4. 临时停车区 → 指挥模式中保持可见直到全部清空
```

### 7.3 共享数据层

```text
以下数据在所有模式之间共享（统一数据层）：
  - 拥堵指数（高德API）
  - 天气数据（气象API）
  - 设备状态（海康平台）
  - 港口数据（港口闸机API）
  - 卡口数据（海康卡口）
  - 预警信息（多源汇聚）

各模式独有的数据：
  指挥模式：归因分析结果、策略执行档案、效果预测值
  应急模式：滞留预测、物资统计、任务板状态、应急预案状态
  
  独有数据在模式切换时不丢失，而是归档保存，
  可在分析模式中回溯查看。
```

## 八、交互流程

### 主流程：台风停航全过程

```text
1. 预警收到（阶段1：预警期）
   系统收到台风预警 / 港口停航提前通知
   → 自动切换应急模式
   → 滞留预测模型启动
   → 任务板自动下发"预警期任务"给各部门
   ↓

2. 指挥部决策
   县领导查看大屏：
     左栏：预计滞留X辆，其中冷链车Y辆
     中央：各临时停车区位置和容量
     右栏：各部门待命状态
   → 确认启动预案（EP-01台风停航预案）
   → 确认应急响应等级
   ↓

3. 停航生效（阶段2：停航初期）
   红色横幅更新："徐闻港已停航"
   → 滞留数实时上涨
   → 任务板自动下发"停航初期任务"
   → 诱导屏自动更新："港口已停航，请勿前往"
   → 交警开始部署
   ↓

4. 持续监控（阶段3：滞留高峰期）
   大屏持续刷新：
     滞留数、物资消耗、冷链车告警
   → 物资缺口时自动生成补给任务
   → 冷链车滞留超阈值时自动告警
   → 无人机定时巡查（如可用）
   ↓

5. 复航准备（阶段4：复航准备期）
   收到港口"即将复航"通知
   → 大屏提示"预计X小时后复航"
   → 自动生成分批放行方案：
     第1批：冷链车（优先）
     第2批：小客车
     第3批：普通货车
   → 任务板下发"复航准备任务"
   ↓

6. 复航消化（阶段5）
   港口恢复运营
   → 分批放行开始
   → 滞留数逐步下降
   → 各部门逐步撤收
   → 满足退出条件 → 切换回总览/指挥模式
   ↓

7. 事件归档
   应急结束后：
   → 完整事件日志自动导出
   → 物资消耗统计报告
   → 各部门响应时效统计
   → 归档到分析模式，供未来复盘和预案优化
```

### Q-02 补充：应急通信方案

```text
应急模式下的跨部门通知方式（替代微信群+电话）：

1. 智能外呼（IP电话模式）
   触发：任务板下发任务时，系统自动拨打对应负责人电话
   内容：TTS语音播报，例如：
     "徐闻县智慧交通应急通知：徐闻港已于14:30停航，
      预计滞留车辆3200辆，请公安交警部门立即启动应急预案，
      部署6组交警至进港大道。请按1确认接收，按2转接值班室。"
   
   确认机制：
     按1 → 系统记录"已接收"，任务板状态更新
     按2 → 转接该部门值班室
     无应答 → 30秒后重拨，3次未接 → 自动拨打备用联系人
     
   技术方案：
     接入云通信平台（如阿里云语音服务/腾讯云呼叫中心）
     成本：约0.1元/分钟，应急场景下月均成本<500元

2. 短信通知（辅助通道）
   触发：与智能外呼同步发送
   内容：精简版任务信息 + 大屏链接
   示例：
     "【徐闻交通应急】徐闻港14:30停航，预计滞留3200辆。
      您的任务：部署6组交警至进港大道。
      请登录大屏系统确认。"
   
   技术方案：
     接入短信网关（阿里云/腾讯云短信服务）
     成本：约0.04元/条

3. 大屏任务板（主通道）
   所有任务的权威来源，外呼和短信是通知手段，
   任务确认和进度更新以大屏任务板为准。

通知优先级：
  Ⅰ级/Ⅱ级应急 → 智能外呼 + 短信 + 大屏（三通道同时）
  Ⅲ级应急 → 短信 + 大屏
  Ⅳ级应急 → 仅大屏通知
```
