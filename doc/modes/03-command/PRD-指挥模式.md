# 03 指挥模式（Command Mode）— 产品需求文档

> 版本：V1.0 | 日期：2026-04-09
> 优先级：P0 | 状态：需求讨论中

---

## 一、模式定位

当拥堵发生或即将发生时，为交通指挥员和县领导提供**拥堵归因分析 + AI策略建议 + 执行跟踪**的完整决策闭环。

核心价值：从"看到拥堵"升级为"知道为什么堵、该怎么办、效果如何"。

---

## 二、触发与退出规则

| 条件 | 行为 |
| ---- | ---- |
| 拥堵指数 > 4（中度拥堵） | 顶部橙色横幅提示，建议手动切换 |
| 拥堵指数 > 6（重度拥堵） | 自动切换到指挥模式，推送通知 |
| 拥堵指数回落 ≤ 2 且持续15分钟 | 提示"拥堵已缓解，是否返回总览模式？" |
| 手动切换 | 任何时候均可从导航栏进入 |

## 四、功能清单

### A. 拥堵态势面板（左栏上部）

实时展示当前拥堵的核心数据，让指挥员10秒内掌握全局。

| 编号 | 功能项 | 说明 | 数据刷新 |
| ---- | ------ | ---- | -------- |
| A-01 | 拥堵等级标签 | 严重拥堵/重度/中度/轻度，颜色区分 | 实时 |
| A-02 | 拥堵指数 | 进港方向 + 出港方向分别展示，保留1位小数 | 5秒 |
| A-03 | 拥堵距离 | 当前拥堵排队长度（米/公里） | 30秒 |
| A-04 | 拥堵持续时间 | 从拥堵开始到当前的持续时长 | 实时 |
| A-05 | 影响车辆数 | 当前受拥堵影响的车辆估算数量 | 1分钟 |
| A-06 | 拥堵趋势箭头 | ↑加剧 / →持平 / ↓缓解，基于近15分钟趋势 | 1分钟 |

### B. 拥堵归因分析（左栏中部）

回答"为什么堵"，这是现有系统完全缺失的核心能力。

| 编号 | 功能项 | 说明 |
| ---- | ------ | ---- |
| B-01 | 归因类型标签 | 系统自动判定拥堵主因，显示归因标签 |
| B-02 | 港口积压型 | 港口运力不足 → 待舶车辆溢出到进港大道 |
| B-03 | 流量高峰型 | 节假日/早晚高峰 → 车流量超过道路承载 |
| B-04 | 事故阻断型 | 交通事故 → 车道被占用 |
| B-05 | 天气影响型 | 大雾/暴雨 → 车速下降/港口限流 |
| B-06 | 施工占道型 | 道路施工 → 可用车道减少（进港大道正在拓宽） |
| B-07 | 复合型归因 | 多因素叠加时，按权重排序展示 |
| B-08 | 归因置信度 | 每个归因标签附带置信度百分比 |

**归因判定逻辑：**

```text
IF 待舶车辆数 > 阈值 AND 港口运力饱和度 > 80%
  → 港口积压型（权重高）

IF 当前车流量 > 历史同时段均值 × 1.5
  → 流量高峰型

IF 近30分钟内有事故报警
  → 事故阻断型

IF 风速 > 10m/s OR 降雨量 > 50mm OR 能见度 < 200m
  → 天气影响型

IF 施工路段信息匹配
  → 施工占道型
```

### C. 影响评估（左栏下部）

量化拥堵的影响范围和严重程度，为决策提供数据支撑。

| 编号 | 功能项 | 说明 |
| ---- | ------ | ---- |
| C-01 | 影响路段数 | 受波及的路段数量 |
| C-02 | 影响车辆总数 | 当前在拥堵区域内的车辆估算 |
| C-03 | 预计持续时长 | 如不干预，拥堵预计还将持续多久 |
| C-04 | 经济损失估算 | 基于车辆数×等待时间的粗略估算（可选） |
| C-05 | 冷链车数量 | 拥堵区域内冷链车数量（特殊关注） |
| C-06 | 波及范围预测 | 如不干预，30分钟后拥堵将扩展到哪些路段 |

| D-06 | 诱导屏位置+内容 | 显示电子诱导屏位置，点击可预览当前发布内容（Mock 数据） |
| D-07 | 卡口实时流量 | 各卡口点位显示实时过车流量数字（Mock 数据） |

### F. AI策略建议（右栏上部）

指挥模式的核心价值——系统主动给出处置建议。

| 编号 | 功能项 | 说明 |
| ---- | ------ | ---- |
| F-01 | 策略卡片 | 每个策略以卡片形式展示，包含名称、描述、预期效果 |
| F-02 | 推荐标记 | 系统推荐的最优方案标记"推荐"标签 |
| F-03 | 策略类型 | 见下方策略库 |
| F-04 | 执行难度评级 | ★~★★★★★，帮助决策者评估可行性 |
| F-05 | 预计生效时间 | 策略执行后多久开始见效 |
| F-06 | 一键执行按钮 | 确认后触发策略执行流程 |
| F-07 | 自定义调整 | 允许修改策略参数后再执行 |

**核心原则：系统提供决策建议，人来做决策。所有策略均需对应权限的人员确认后方可执行。**


## 十二、盲区补充（2026-04-13）

### 12.1 完整策略冲突矩阵

15个策略之间的冲突关系分为三类：互斥（不可同时执行）、约束（可同时但需参数协调）、联动（一个触发自动影响另一个）。

#### 互斥关系

| 策略A | 策略B | 互斥原因 |
| ----- | ----- | -------- |
| S-01 应急车道借用 | S-13 潮汐车道借用 | 两者同时执行意味着进港方向占用5车道（原3+应急1+借用1），出港仅剩1车道+无应急车道，安全风险不可接受 |
| S-06 限流入港 | S-15 预约通行引导（限流期间） | 限流时预约车辆享有优先放行权，两者不互斥但需协调：预约车辆从限流批次中优先放行，非预约车辆排后 |
| S-11 货客分时（限货时段） | S-12 冷链车优先 | 不互斥，但需明确：冷链车属于S-11的豁免车辆，S-12的优先通道是在S-11基础上的增强措施 |

#### 约束关系（可并行，需参数协调）

| 策略组合 | 约束规则 |
| -------- | -------- |
| S-02 + S-03 | 总分流比例 ≤ 60%，推荐比例 S376:G207 = 6:4 |
| S-01 + S-13 | **互斥**，见上方 |
| S-04 + S-02/S-03 | 信号灯配时需考虑分流后的流量变化：分流启动后15分钟，信号灯应从拥堵配时逐步回调（因为流量已被分流降低） |
| S-06 限流 + S-04 信号灯 | 限流后进港大道流量下降，信号灯配时应同步回调，否则进港方向绿灯过长造成交叉方向不必要延误 |
| S-06 限流 + S-08 临时停车区 | 限流导致城区路口排队超过3个路口时，必须同步启用S-08，引导被限流车辆进入临时停车区等待 |
| S-11 货客分时 + S-02/S-03 分流 | 分时限货后进港大道流量结构变化（客车占比升高），分流比例需重新计算：客车对诱导屏分流建议的响应率通常低于货车（约60% vs 80%），实际分流效果打折 |

#### 联动关系（自动触发）

| 触发事件 | 自动联动 |
| -------- | -------- |
| S-07 事故在G207 | 自动禁用 S-03（G207分流），标记"不可用：目标路段有事故" |
| S-07 事故在S376 | 自动禁用 S-02（S376分流） |
| S-07 事故在进港大道 | 自动启用 S-02 + S-03 分流建议 |
| S-02/S-03 分流启动 | 自动触发 S-09 诱导屏发布分流信息 |
| S-06 限流启动 | 自动检查城区排队长度，超过3个路口则建议启用 S-08 |
| S-01 应急车道借用 | 自动禁用 S-13 潮汐车道借用的建议 |
| 任何策略执行15分钟后效果不足 | 系统自动建议升级到下一阶段策略组合 |

#### 施工期间的特殊约束

进港大道正在拓宽施工，可用车道数动态变化：

```text
施工期间车道情况（需实时确认）：
  正常：双向6车道（进3出3）
  施工中：可能降至双向4车道（进2出2）

施工对策略的影响：
  S-01 应急车道借用：
    正常时 → 进4出3（可行）
    施工时 → 进3出2（可行但余量小）
  
  S-13 潮汐车道借用：
    正常时 → 进4出2（可行，出港保留2车道）
    施工时 → 进3出1（⚠️ 高风险，出港仅1车道，建议禁用）
  
  系统需维护"当前施工状态"字段：
    施工中 = true → S-13自动标记为"不可用：施工期间出港车道不足"
    施工中 = false → S-13恢复可用
```

### 12.2 归因分析的数据延迟处理

海康卡口数据（1小时粒度）与高德API（5秒级）的时间粒度差异很大，归因逻辑需要明确如何处理。

#### 数据源时间粒度对照

| 数据源 | 刷新频率 | 用于归因的场景 |
| ------ | -------- | -------------- |
| 高德API | 5秒 | 拥堵指数、拥堵距离、拥堵时间、路段车速 |
| 海康卡口 | 5分钟（推动后）/ 1小时（现状） | 各卡口过车流量、车型识别 |
| 港口闸机 | 5分钟（推动后） | 待舶车辆数、进出港车辆数 |
| 气象API | 10分钟 | 风速、降雨量、能见度 |
| 公安系统 | 事件触发 | 事故报警 |
| 施工信息 | 每日更新 | 施工路段、占道情况 |

#### 归因判定的时间窗口策略

```text
归因引擎采用"多时间窗口融合"策略：

快速归因（5秒内，基于高德数据）：
  拥堵指数突然飙升 + 某路段车速骤降至0
  → 初步判定"疑似事故阻断型"（置信度60%）
  → 立即在大屏显示，标注"待确认"

中速归因（5分钟内，融合卡口+港口数据）：
  卡口数据确认流量异常 + 港口待舶数据确认积压
  → 修正归因为"港口积压型"（置信度85%）
  → 更新大屏显示，去掉"待确认"标注

完整归因（15分钟内，融合所有数据源）：
  叠加气象数据 + 施工信息 + 历史同期对比
  → 最终归因"港口积压型（主因70%）+ 天气影响型（次因30%）"
  → 置信度提升至90%+

关键设计原则：
  1. 宁可先给一个粗略归因，也不要等15分钟才给结果
  2. 归因结果是"渐进式精化"的，大屏上显示置信度让指挥员知道可靠程度
  3. 策略建议在"快速归因"阶段就可以生成（基于初步判定），
     但标注"基于初步分析，建议等待确认后执行"
  4. 高风险策略（🔴级）必须等到"中速归因"完成后才允许提交审批
```

#### 海康卡口数据延迟的降级方案

### 12.3 效果预测模型的冷启动方案

初期没有历史策略执行数据，效果预测值从哪来？

#### 三阶段方案

```text
阶段一：专家经验值（上线即可用，0数据量）
─────────────────────────────────────────
  每个策略的效果预测基于交通工程经验值 + 徐闻本地道路参数：

  S-01 应急车道借用：
    通行能力提升 = 新增1车道 ÷ 原车道数 = 1/3 ≈ 33%
    拥堵指数预计降低 = 当前指数 × 0.33 × 效率折扣0.7 ≈ 当前指数 × 0.23
    （效率折扣：应急车道窄、限速40、汇入摩擦）
    缓解时间 = 排队长度(m) ÷ (新增通行能力 × 平均车间距) + 10分钟缓冲

  S-02 S376分流：
    分流比例30% → 进港大道流量降低30% × 实际响应率60% = 实际降低18%
    拥堵指数预计降低 = 当前指数 × 0.18
    缓解时间 = 诱导屏发布后20分钟（司机看到→决策→绕行→生效）

  S-04 信号灯配时优化：
    进港方向绿灯+25% → 通过率提升约15%（非线性，受饱和度影响）
    拥堵指数预计降低 = 当前指数 × 0.15 × 路口数权重
    缓解时间 = 3-5分钟（信号灯调整即时生效）

  大屏显示：
    效果预测值后标注"（基于经验模型）"
    置信区间较宽：±30%

阶段二：校准修正（10-50次执行数据，约1-3个月）
─────────────────────────────────────────
  每次策略执行后，系统自动对比：
    预测效果 vs 实际效果 → 计算偏差率
  
  校准方法：
    修正系数 = 近N次实际效果均值 ÷ 近N次预测效果均值
    新预测值 = 经验值 × 修正系数
  
  示例：
    S-02分流预测降低18%，实际5次执行平均降低14%
    修正系数 = 14/18 = 0.78
    后续预测 = 经验值 × 0.78
  
  大屏显示：
    效果预测值后标注"（基于经验模型+10次校准）"
    置信区间收窄：±20%

阶段三：ML模型（50+次执行数据，约6个月后）
─────────────────────────────────────────
  训练回归模型：
    输入特征：当前拥堵指数、排队长度、时段、天气、港口状态、车型分布
    输出：策略执行后的拥堵指数变化曲线
  
  模型选型：
    初期：XGBoost/LightGBM（可解释性好，数据量要求低）
    后期：如数据量充足，可尝试时序模型（LSTM）
  
  大屏显示：
    效果预测值后标注"（AI预测模型）"
    置信区间：±10%
```

#### 冷链车识别的增强方案

```text
冷链车是应急场景中最关键的特殊车辆（不能熄火），但识别难度最大。

方案一：多源数据融合（推荐，成本低）
  数据源1：运输企业报备数据库（覆盖率约70%的冷链车）
  数据源2：港口售票系统（购票时需填写车辆类型，覆盖率接近100%）
  数据源3：卡口车牌识别 → 匹配报备数据库
  
  融合逻辑：
    车牌命中报备数据库 → 确认冷链车（置信度95%）
    车牌命中售票系统"冷链"标签 → 确认冷链车（置信度98%）
    车牌未命中但黄牌+车身尺寸匹配货车 → 标记"疑似货车，待确认"
  
  综合识别率预估：85-90%（报备70% + 售票补充15-20%）

方案二：AI视觉增强（成本中等，可后期追加）
  在关键卡口（进港大道入口、城区路口）部署AI识别算法：
    特征：车顶/车头制冷机组、车身保温层厚度、排气管位置
    训练数据：从已确认的冷链车图片中训练
    预计增加识别率：+5-8%
  
  成本估算：
    AI算法开发：约10-15万
    边缘计算设备（如已有海康AI盒子则无需额外采购）：0-5万

方案三：主动申报（零成本，但依赖司机配合）
  在预约购票环节增加"冷链车"勾选项
  在进港大道入口设置"冷链车请走右侧车道"标识
  配合交警现场确认

推荐组合：方案一（立即实施）+ 方案三（同步实施）+ 方案二（后期追加）
预计综合识别率：90-95%
```

#### 锂电池类车辆的识别难题

```text
锂电池/铅酸电池类货车是最难识别的车辆类型：
  - 外观与普通货车无明显区别
  - 报备数据覆盖率仅约60%
  - 但这类车辆必须走海安新港，是硬性规定

解决方案：
  1. 港口售票系统是最可靠的数据源
     购票时必须申报货物类型，锂电池类会被系统自动分配到海安新港
     → 如果售票系统数据可接入，识别率接近100%
  
  2. 卡口兜底检测
     如果已购徐闻港船票的车辆被卡口识别为"报备数据库中的锂电池运输车"
     → 系统自动告警："⚠️ 疑似锂电池类车辆驶向徐闻港，请核实"
     → 通知现场交警拦截确认
  
  3. 漏检风险评估
     最坏情况：未报备+未在售票系统标注 → 漏检
     但这类车辆到达港口后，港口安检环节会再次检查
     → 大屏系统的识别是"提前预警"，不是唯一防线
     
  结论：大屏系统定位为"尽力识别+提前预警"，
        港口安检是最终防线，两者互补而非替代
```
