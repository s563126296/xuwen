# 指挥模式 — 指标体系设计

> 版本：V1.0 | 日期：2026-04-14
> 作者：张明晨 | 公司：科大讯飞
> 基于：PRD-指挥模式 V1.0 + 策略执行细则 + 总览模式指标体系设计经验
> 核心变化：从「41 个指标平铺」重构为「决策链 4 层架构」

---

## 一、设计理念

### 1.1 指挥模式的本质

总览模式回答的是「正不正常」，指挥模式回答的是：

> **为什么堵？该怎么办？效果如何？**

这是一条完整的决策链：

```
感知拥堵 → 分析原因 → 制定策略 → 预测效果 → 执行策略 → 验证效果
  看到         想明白       拿主意       有把握       干起来       看结果
```

值班指挥员的思维过程就 6 步。指标体系必须严格对应这 6 步，不能多也不能少。

### 1.2 与总览模式的关系

总览模式是「体检报告」— 扫一眼知道全身状况。
指挥模式是「手术台」— 针对具体问题精准处置。

两者共享基础数据（拥堵指数、港口数据、气象数据），但指挥模式多了三个总览模式没有的能力：
- **归因分析**：不只知道堵，还知道为什么堵
- **策略推荐**：不只看到问题，还给解决方案
- **闭环验证**：不只执行策略，还验证效果

### 1.3 指标分层架构

```
┌─────────────────────────────────────────────────┐
│ 第一层：指挥摘要（1 句话结论 + 推荐动作）        │
├─────────────────────────────────────────────────┤
│ 第二层：决策链指标（6 步决策对应 6 组指标）       │
│   ① 态势感知  ② 归因分析  ③ 影响评估            │
│   ④ 策略推荐  ⑤ 效果预测  ⑥ 执行跟踪            │
├─────────────────────────────────────────────────┤
│ 第三层：辅助信息（按需展开）                      │
│   港口实时 · 气象 · 设备状态 · 历史效果           │
├─────────────────────────────────────────────────┤
│ 第四层：原始数据源（不直接展示）                  │
│   高德 API · 海康卡口 · 港口闸机 · 气象 API      │
└─────────────────────────────────────────────────┘
```

## 二、第一层：指挥摘要

### 2.1 指挥摘要条

**位置**：顶部导航栏下方，替代总览模式的 AI 态势摘要

**核心区别**：总览摘要是「观察者视角」（告诉你情况），指挥摘要是「指挥官视角」（告诉你该怎么做）

**内容结构**：
```
[拥堵等级图标] 拥堵概况一句话 | 推荐策略 | 预计缓解时间
```

**示例**：
```
🔴 进港大道重度拥堵（指数 6.8）| 建议：S376 分流 + 应急车道借用 | 预计 45 分钟缓解
```
```
🟡 进港大道中度拥堵（指数 4.2）| 建议：信号灯优化 + 诱导屏发布 | 预计 20 分钟缓解
```
```
🟢 拥堵已缓解（指数 1.8，持续 12 分钟）| 建议返回总览模式
```

**分级规则**：

| 拥堵指数 | 等级 | 颜色 | 摘要模板 |
|---------|------|------|---------|
| 2-4 | 轻度 | 黄色 | "{路段}轻度拥堵（指数 {X}），{归因}，建议{首选策略}" |
| 4-6 | 中度 | 橙色 | "{路段}中度拥堵（指数 {X}），{归因}，建议{策略组合}，预计{Y}分钟缓解" |
| 6-8 | 重度 | 红色 | "{路段}重度拥堵（指数 {X}），{归因}，建议{策略组合}，需{审批级别}审批" |
| >8 | 严重 | 深红 | "⚠️ {路段}严重拥堵（指数 {X}），建议全量策略启动，请值班领导决策" |

**数据依赖**：T-01/T-02（拥堵指数）、AI-01/AI-02（归因）、AI-03（推荐策略）、AI-06（预测效果）

### 3.2 归因分析（回答：为什么堵？）

**对应 PRD 模块**：B 模块

**核心设计**：这是指挥模式最核心的差异化指标。市面上大多数交通大屏只能告诉你「哪里堵」，我们要告诉你「为什么堵」。

**5 种归因类型**：

| 归因类型 | 判定规则 | 图标 | 典型表现 |
|---------|---------|------|---------|
| 港口积压型 | 待舶车辆 >500 且排队向上游蔓延 | 🚢 船 | 港口排队溢出到进港大道 |
| 流量高峰型 | 当前流量 >历史同期均值 1.5 倍 | 📈 趋势 | 下班高峰 + 港口大船靠港叠加 |
| 事故阻断型 | 公安系统事故报警 + 流速骤降 | ⚠️ 三角 | 追尾/刮蹭导致车道占用 |
| 天气影响型 | 降雨 >10mm/h 或能见度 <200m 或风力 >7 级 | 🌧️ 天气 | 暴雨导致车速下降、能见度低 |
| 施工占道型 | 施工信息库匹配 + 该路段拥堵 | 🚧 施工 | 道路拓宽施工占用车道 |

**多因叠加**：支持多种归因同时显示，按置信度排序

**显示格式**：
```
拥堵归因分析
├── 港口积压型（置信度 85%）★ 主因
│   └── 徐闻港待舶 680 辆，排队溢出至进港大道 S289 交叉口
├── 流量高峰型（置信度 60%）
│   └── 当前流量为历史同期 1.8 倍（周五下午 + 大船靠港）
└── 天气影响型（置信度 20%）
    └── 阵雨（降雨量 5mm/h），影响较小
```

**归因判定引擎（规则 V1）**：

```
港口积压型判定：
  IF P-01(待舶) > 500 AND T-07(排队) > 1000m
  AND 排队尾部坐标超出港口范围
  THEN 归因 = 港口积压型, 置信度 = min(P-01/500, T-07/1000) × 85%

流量高峰型判定：
  IF T-05(当前流量) > 历史同期均值 × 1.5
  THEN 归因 = 流量高峰型, 置信度 = (当前/均值 - 1) × 60%

事故阻断型判定：
  IF E-01(事故数) > 0 AND 事故路段 T-06(车速) < 10km/h
  THEN 归因 = 事故阻断型, 置信度 = 90%

天气影响型判定：
  IF W-02(降雨) > 10 OR W-03(能见度) < 200 OR W-01(风速) > 17
  THEN 归因 = 天气影响型, 置信度 = max(降雨/25, 200/能见度, 风速/25) × 70%

施工占道型判定：
  IF E-03(施工路段) 匹配当前拥堵路段
  THEN 归因 = 施工占道型, 置信度 = 75%
```

**冷启动方案**：
- 第 1-4 周：纯规则匹配，置信度标注「规则推断」
- 第 5-12 周：积累 20+ 案例后，用历史结果校准规则阈值
- 第 13 周+：如数据充分，训练分类模型替代规则

**数据依赖**：AI-01, AI-02, P-01, T-05, T-06, T-07, E-01, E-03, W-01~W-03

### 3.3 影响评估（回答：影响有多大？）

**对应 PRD 模块**：C 模块

**指标清单**：

| 指标 | 计算方式 | 显示方式 |
|------|---------|---------|
| 影响路段数 | 拥堵指数 >2 的路段计数 | 数字 |
| 影响车辆数 | 入口流量 - 出口流量（卡口差值） | 数字 + 趋势 |
| 持续时长 | 拥堵指数首次 >4 起计时 | 计时器 |
| 冷链车数量 | P-10 车辆类型分布中冷链占比 × 影响车辆 | 数字 + 警告标记（>20 辆高亮）|
| 波及预测 | 当前传导速率 × 时间 = 未来 30 分钟新增影响路段 | 文字描述 |

**可视化设计**：
- 紧凑型卡片，在归因分析下方
- 关键数字用色块背景突出
- 冷链车/危化品特殊标记（涉及食品安全/公共安全）

**与总览模式的数据复用**：
- 影响评估可直接读取总览模式的「拥堵扩散情况」（PressureTransmission）数据
- 新增：冷链车识别（需 P-10 港口售票数据）

### 3.4 策略推荐（回答：该怎么办？）

**对应 PRD 模块**：F 模块

**核心设计**：不给用户 15 个策略让他自己选，而是根据归因类型自动推荐最佳组合。

**策略推荐引擎（规则 V1）**：

| 归因类型 | 推荐策略组合 | 审批要求 |
|---------|------------|---------|
| 港口积压（轻，指数 3-4） | S-04 信号优化 + S-09 诱导屏 | 🟡 指挥员确认 |
| 港口积压（中，指数 4-6） | S-02 S376 分流 + S-04 + S-14 交警 + S-09 | 🟡 指挥员确认 |
| 港口积压（重，指数 6-8） | S-01 应急车道 + S-02 + S-03 G207 + S-06 限流 + S-05 增班 | 🔴 领导审批 |
| 港口积压（极重，指数 >8） | 全量策略 + S-13 潮汐车道 + S-08 临时停车 | 🔴 领导 + 🔵 跨部门 |
| 流量高峰型 | S-04 + S-02 + S-14 | 🟡 指挥员确认 |
| 事故阻断型 | S-07 事故处置 + S-09 + 事故路段避让分流 | 🟡 指挥员确认 |
| 天气影响型 | S-09 诱导屏 + S-04 降速信号 | 🟡 指挥员确认 |
| 施工占道型 | S-02/S-03 绕行 + S-09 | 🟡 指挥员确认 |
| 多因叠加 | 按主因推荐 + 叠加次因补充策略 | 取最高审批级别 |

**策略卡片显示**：

```
┌─ 推荐策略 ────────────────────────────────┐
│                                            │
│  ★ S-02 S376 省道分流              🟡 确认  │
│  分流比例 30% | 生效时间 ~20 分钟           │
│  预计拥堵指数降低 1.2                       │
│  ─────────────────────────────────────     │
│                                            │
│  S-04 信号灯优化（方案 B）          🟡 确认  │
│  进港方向绿灯延长 15 秒 | 生效 ~5 分钟      │
│  预计通过能力提升 15%                       │
│  ─────────────────────────────────────     │
│                                            │
│  S-14 交警增援（2 组）             🟡 确认   │
│  部署至进港大道/S289 路口 | 到位 ~15 分钟    │
│                                            │
│  [一键执行全部] [逐个确认] [查看更多策略]    │
└────────────────────────────────────────────┘
```

**策略冲突检查**：
- 执行前自动检查互斥关系（S-01 vs S-13 不可同时）
- 约束关系自动协调（S-02 + S-03 总分流 ≤60%）
- 联动策略自动触发（分流启动 → 自动发诱导屏 S-09）

### 3.5 效果预测（回答：做了会怎样？）

**对应 PRD 模块**：G 模块

**核心指标**：

| 指标 | 说明 | 显示方式 |
|------|------|---------|
| 预测拥堵指数变化 | 执行策略后的拥堵指数曲线 | 折线图（当前 → 30 分钟后） |
| 预计缓解时间 | 拥堵指数降到目标值（≤3）的预估时间 | 大字数字 + 置信区间 |
| 预计减少排队 | 排队长度变化预测 | 数字（如 2.5km → 0.8km） |
| 多方案对比 | 不同策略组合的效果对比 | 并列柱状图 |
| 风险提示 | 策略可能的副作用 | 文字警告（如「S376 分流可能导致本地交通受影响」） |

**预测模型（三阶段）**：

| 阶段 | 条件 | 方法 | 置信度标注 |
|------|------|------|----------|
| V1 经验值 | 0 数据即可用 | 策略效果 = 当前指数 × 经验系数 | 「基于经验模型，误差 ±30%」 |
| V2 校准修正 | 10-50 次执行 | 经验值 × 校准系数（实际/预测） | 「基于 N 次校准，误差 ±20%」 |
| V3 ML 模型 | 50+ 次执行 | XGBoost 回归（输入：多维状态 → 输出：效果曲线） | 「AI 预测，准确率 X%」 |

**经验系数表（V1 阶段）**：

| 策略 | 预期效果 | 生效时间 | 效率折扣 |
|------|---------|---------|---------|
| S-01 应急车道 | 通行能力 +33% | 5 分钟 | ×0.7（实际 +23%） |
| S-02 S376 分流 | 主路流量 -30% | 20 分钟 | ×0.8（实际 -24%） |
| S-03 G207 分流 | 主路流量 -20% | 25 分钟 | ×0.75（实际 -15%） |
| S-04 信号灯优化 | 通过能力 +15% | 5 分钟 | ×0.85（实际 +13%） |
| S-06 限流入港 | 入港流量 -40% | 即时 | ×0.9（实际 -36%） |
| S-07 事故处置 | 恢复通行 | 15-45 分钟 | 取决于事故类型 |
| S-14 交警增援 | 通过效率 +10% | 15 分钟 | ×0.8（实际 +8%） |

**可视化设计**：
- 右栏中部卡片
- 折线图展示预测曲线（虚线）vs 当前趋势（实线）
- 策略执行时间点标注在曲线上
- 置信区间用浅色带表示

### 3.6 执行跟踪（回答：做到哪了？效果怎样？）

**对应 PRD 模块**：H 模块

**执行状态机**：

```
待审批 → 已审批 → 执行中 → 观察期 → 达标/未达标
  │                                      │
  └── 被驳回                              └── 未达标 → 追加策略建议
```

**核心指标**：

| 指标 | 说明 | 显示方式 |
|------|------|---------|
| 执行状态 | 每个策略的当前状态 | 状态标签 + 进度条 |
| 执行时长 | 策略启动后经过的时间 | 计时器 |
| 预测 vs 实际 | 预测拥堵指数 vs 实际拥堵指数 | 双线对比折线图 |
| 达标判定 | 实际效果是否达到预测的 70% | 达标/未达标标签 |
| 追加建议 | 15 分钟未达标时的升级策略建议 | 弹出卡片 |

**达标判定规则**：
- 执行后 15 分钟：实际降幅 ≥ 预测降幅 × 70% → 达标
- 执行后 15 分钟：实际降幅 < 预测降幅 × 50% → 未达标，建议追加策略
- 执行后 15 分钟：实际降幅在 50%-70% 之间 → 继续观察，延长至 30 分钟判定

**可视化设计**：
- 右栏底部，可纵向展开
- 每个策略一行：[策略名] [状态标签] [进度条] [效果指标]
- 未达标时红色高亮 + 追加策略建议按钮

```
                   └──────┬──────┘
                          │ 对比
                   ┌──────▼──────┐
                   │ ⑥ 执行跟踪   │
                   │ 做到哪了？   │
                   └──────┬──────┘
                          │ 反馈
                          ▼
                   未达标 → 回到 ④ 追加策略
                   达标 → 继续观察 → 退出指挥模式
```

**数据流向**：
- ①→② 拥堵数据触发归因分析
- ②→④ 归因类型决定策略推荐
- ④→⑤ 策略参数输入预测模型
- ⑤→⑥ 预测值作为执行效果的对照基准
- ⑥→④ 未达标时反馈到策略推荐，触发升级

## 七、与总览模式的数据复用

| 总览模式指标 | 指挥模式复用方式 |
|------------|----------------|
| 拥堵扩散情况（PressureTransmission） | 影响评估的波及预测基础 |

| 进出港流量对比（TidalEffect） | 归因判定的输入（潮汐方向） |
| 道路剩余容量（CorridorElasticity） | 策略推荐的约束条件（分流路容量） |
| 天气影响程度（WeatherCoupling） | 归因判定的输入 |
| 停航风险预测（ShutdownProbability） | 策略推荐的上下文 |
| AI 态势摘要（AiSummary） | 替换为指挥摘要（不同模板） |

## 八、实施优先级

### 第一批：数据驱动的骨架（1-2 周）

目标：现有 UI 骨架改为从 store 读数据，store 用完整 Mock。

- [ ] dashboardStore 扩展指挥模式状态切片（归因、策略列表、执行状态、预测数据）
- [ ] 态势感知卡片（读 store 拥堵数据，大字号显示）
- [ ] 归因分析卡片（规则引擎 V1，基于 store 数据自动判定）
- [ ] 影响评估卡片（读 store + 计算）
- [ ] 指挥摘要条（模板生成）
- [ ] 时间轴改为读 store 历史 + 预测数据

### 第二批：策略推荐和预测（2-3 周）

目标：策略引擎上线，从「看得清」到「判得准」。

- [ ] 策略推荐引擎（归因类型 → 策略匹配 → 冲突检查）
- [ ] 策略卡片交互（展开详情、参数调整、确认/驳回）
- [ ] 效果预测 V1（经验系数模型）
- [ ] 效果预测曲线可视化
- [ ] 地图增加拥堵着色 + 分流路线高亮

### 第三批：执行闭环（2-3 周）

目标：策略执行到验证的完整闭环。

- [ ] 策略执行状态机（审批 → 执行 → 观察 → 判定）
- [ ] 执行跟踪面板（预测 vs 实际对比）
- [ ] 未达标时自动建议追加策略
- [ ] 历史效果统计面板
- [ ] 策略升级/退出路径
- [ ] 策略执行档案自动记录
