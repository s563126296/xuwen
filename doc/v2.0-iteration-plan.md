## 一、核心需求变更

### 1.1 应急与指挥模式合并

**业务背景：**
- 应急业务过于复杂，实际操作中系统难以完全覆盖
- 业务上强调"指挥"，轻"应急"
- 需要简化应急流程，整合到指挥模式

**技术方案：**

**方案 A：指挥模式吸收应急功能（推荐）**
```
新指挥模式 = 原指挥模式 + 应急核心功能

保留：
- 指挥模式的 6 步决策链
- 策略推荐和冲突检测
- 执行资源面板
- 视频通信系统

整合：
- 应急预案数字化执行（简化版）
- 特殊车辆追踪（冷链车/危化品车）
- 物资需求估算（简化版）
- 跨部门任务板（整合到执行资源）

移除：
- 复杂的 5 阶段状态机
- 台风路径动画
- 滞留预测模型
- 视频会商系统（保留基础通信）
```

**方案 B：创建统一指挥中心模式**
```
新建"指挥中心"模式，替代原指挥+应急

特点：
- 统一的指挥界面
- 根据事件类型（拥堵/台风/事故）动态调整面板
- 轻量级应急响应流程
- 保留指挥决策的核心能力
```

**推荐：方案 A**
- 改动成本较低
- 保留指挥模式的成熟架构
- 应急功能作为"增强模块"整合

**工作量评估：**
- 需求分析与 PRD 重写：2-3 天
- 代码重构与整合：5-7 天
- 测试与调试：2-3 天
- 文档更新：1-2 天
- **总计：10-15 天**

### 1.2 突出 AI 能力展示

**需求描述：**
在总览和指挥模式下增强 AI 能力的可见性和实用性。

**总览模式 AI 增强：**

1. **AI 态势研判升级**
   - 当前：静态规则引擎（9 个指标 → 告警等级）
   - 升级：动态预测模型
     - 未来 30 分钟/1 小时态势预测
     - 异常模式识别（节假日/台风/事故）
     - 主动预警（拥堵指数即将超阈值）

2. **AI 建议更主动**
   - 当前：被动显示建议
   - 升级：主动推送建议
     - 检测到异常立即弹出建议卡片
     - 语音播报（配合虚拟人）
     - 建议优先级排序

3. **AI 学习能力展示**
   - 显示"AI 已学习 X 个历史案例"
   - 显示"预测准确率 X%"
   - 显示"本周 AI 建议采纳率 X%"

**指挥模式 AI 增强：**

1. **智能归因分析**
   - 当前：规则匹配（港口积压型/流量高峰型）
   - 升级：多因素权重分析
     - 显示各因素贡献度（港口 60% + 流量 30% + 天气 10%）
     - 历史相似案例对比
     - 根因分析（为什么港口积压？）

2. **策略推荐优化**
   - 当前：固定策略库匹配
   - 升级：动态策略组合
     - 根据历史效果自动调整推荐顺序
     - 显示"该策略在类似场景下成功率 X%"
     - 策略组合优化（S-02+S-04 比单独执行效果好 20%）

3. **效果预测可视化**
   - 当前：简单的预测曲线
   - 升级：置信区间 + 风险提示
     - 显示"预测缓解时间 25-35 分钟（置信度 85%）"
     - 显示"如果 X 条件变化，缓解时间可能延长至 Y"

**技术实现：**
- 前端：增强可视化组件（置信区间、权重饼图、时间线预测）
- 后端：Mock 数据模拟 AI 模型输出（真实 AI 模型需要数据积累）
- 文案：强调"AI 分析"、"智能推荐"、"机器学习"等关键词

**工作量评估：**
- 总览模式 AI 增强：3-4 天
- 指挥模式 AI 增强：4-5 天
- **总计：7-9 天**

### 1.3 总览模式增加虚拟人功能

**需求描述：**
在总览模式增加虚拟人，支持主动播报和对话交互。

**功能设计：**

**1. 虚拟人形象**
```
位置：总览模式右下角（或左下角）
尺寸：300x400px 浮动窗口
形象：
  - 方案 A：2D 卡通形象（成本低，推荐）
  - 方案 B：3D 数字人（成本高，效果好）
  - 方案 C：静态头像 + 动态波形（最简单）
```

**2. 主动播报**
```
触发场景：
- 系统启动时："早上好，当前交通态势正常，拥堵指数 2.8"
- 异常检测时："检测到进港大道拥堵指数升至 4.5，建议切换指挥模式"
- 定时播报：每小时整点播报一次态势摘要
- 模式切换时："已切换到指挥模式，当前拥堵指数 6.2"

播报内容：
- 从 AI 摘要条提取关键信息
- 简洁明了，30 秒以内
- 语音合成（TTS）

技术方案：
- TTS 引擎：讯飞语音 / 阿里云 TTS / 浏览器 Web Speech API
- 播报队列：避免多条消息同时播报
- 静音控制：用户可关闭语音
```

**3. 对话交互**
```
交互方式：
- 点击虚拟人 → 弹出对话框
- 语音输入（可选）或文字输入
- 虚拟人语音回复 + 文字显示

支持的问题类型：
- 态势查询："当前拥堵情况怎么样？"
- 数据查询："今天进港车辆有多少？"
- 功能引导："怎么切换到指挥模式？"
- 历史查询："上周五发生了什么？"

技术方案：
- 方案 A：规则匹配（关键词识别 + 模板回复）
  - 成本低，响应快
  - 覆盖 20-30 个常见问题
  - 推荐用于 v2.0
  
- 方案 B：接入大模型（GPT/Claude/讯飞星火）
  - 成本高，需要 API
  - 理解能力强，可以回答复杂问题
  - 适合后续迭代

- 方案 C：混合方案
  - 常见问题用规则匹配
  - 复杂问题转接大模型
  - 平衡成本和效果
```

**4. UI 设计**
```
虚拟人窗口：
- 默认状态：小窗口，显示头像 + 状态指示灯
- 播报状态：头像动画 + 语音波形
- 对话状态：展开对话框，显示历史消息

交互细节：
- 点击头像：展开/收起对话框
- 点击静音按钮：关闭语音播报
- 拖拽：调整窗口位置
- 最小化：缩小到角落
```

**技术实现：**

**前端组件：**
```typescript
// 虚拟人主组件
<VirtualAssistant />
  - <AvatarWindow />      // 头像窗口
  - <ChatDialog />        // 对话框
  - <VoiceWaveform />     // 语音波形
  - <TTSController />     // TTS 控制器

// 状态管理
interface AssistantState {
  isVisible: boolean;
  isSpeaking: boolean;
  isChatOpen: boolean;
  messages: Message[];
  voiceEnabled: boolean;
}
```

**TTS 集成：**
```typescript
// 使用讯飞语音 TTS（推荐）
import { XFVoice } from '@xfyun/voice-sdk';

// 或使用浏览器 Web Speech API（免费）
const synth = window.speechSynthesis;
const utterance = new SpeechSynthesisUtterance(text);
utterance.lang = 'zh-CN';
synth.speak(utterance);
```

**对话引擎（规则匹配）：**
```typescript
const qaRules = [
  {
    keywords: ['拥堵', '堵车', '交通'],
    response: () => `当前拥堵指数 ${state.congestionIndex}，${state.aiSummary}`
  },
  {
    keywords: ['车辆', '车流'],
    response: () => `今日进港车辆 ${state.todayVehicles} 辆`
  },
  // ... 20-30 条规则
];
```

**工作量评估：**
- 虚拟人 UI 组件：2-3 天
- TTS 集成与播报逻辑：2-3 天
- 对话引擎（规则匹配）：3-4 天
- 测试与优化：2 天
- **总计：9-12 天**

## 二、技术方案总结

### 2.1 架构调整

**模式变更：**
```
v1.0: 5 个模式（总览/指挥/应急/港口/分析）
v2.0: 4 个模式（总览/指挥/港口/分析）

变更：
- 应急模式 → 整合到指挥模式
- 指挥模式 → 增强版指挥模式（含应急能力）
```

**组件复用：**
```
从应急模式迁移到指挥模式：
✅ SpecialVehiclePanel（特殊车辆追踪）
✅ SupplyDemandPanel（物资需求，简化版）
✅ EmergencyTaskBoard（整合到 ExecutionResourcePanel）
✅ EmergencyPlanPanel（简化为预案快捷入口）

移除：
❌ EmergencySimulator（复杂的 5 阶段状态机）
❌ EmergencyVideoConference（保留基础通信）
❌ EmergencyForecastPanel（滞留预测模型）
```

## 四、风险与挑战

### 4.1 技术风险

**应急指挥合并：**
- 风险：功能整合可能导致界面过于复杂
- 缓解：保持指挥模式的简洁性，应急功能作为"可选面板"

**虚拟人 TTS：**
- 风险：浏览器 TTS 质量参差不齐，部分浏览器不支持中文
- 缓解：优先使用讯飞 TTS，浏览器 TTS 作为降级方案

**对话引擎：**
- 风险：规则匹配覆盖有限，无法回答复杂问题
- 缓解：明确告知用户"我只能回答常见问题"，提供问题示例

### 4.2 业务风险

**应急功能简化：**
- 风险：简化后可能无法满足真实应急场景需求
- 缓解：与项目经理确认保留的核心功能清单

**AI 能力展示：**
- 风险：当前是 Mock 数据，真实 AI 模型需要数据积累
- 缓解：文案上强调"演示版本"，真实部署后需要训练模型
