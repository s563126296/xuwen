# 第二章 现状及需求分析

## 2.1 现状分析

### 2.1.1 徐闻县交通区位概况

徐闻县位于中国大陆最南端，是琼州海峡北岸的重要交通枢纽，承担着粤琼两省陆路交通的咽喉功能。徐闻港（新港）作为琼州海峡主要客货滚装运输港口，是进出海南岛的核心通道之一。根据2026年春运公开报道，徐闻港（含海安新港、海安港应急泊位）40天内完成航班吞吐量超1.1万个、旅客吞吐量约350万人、车辆吞吐量约100万辆，春运期间日均保障约8.6万旅客、2.4万车辆过海，单日航班、车辆、货物吞吐量均创历史新高，充分体现了琼州海峡通道在节假日和集中出行场景下的高强度交通保障压力。

**交通基本数据：**

- 日均进港车辆：约8,000辆（平日），春运高峰期进港方向日均车流量可达50,000辆次
- 港口日均发船班次：约100+班次（徐闻港约86班/日 + 海安新港约21班/日）
- 单船载车量：35-120辆（主力船型60-80辆，大型船可达120辆以上）
- 港口日消化能力：约8,000-10,000辆（受泊位、航次、单船载车量限制）
- 主要进港通道：进港大道（双车道至三车道，约14.8公里）、徐海路（三车道，约4.2公里）、环半岛公路（三车道，约7.3公里），三条道路合计约20公里以上
- 拥堵高峰时段：中午至晚上八九点
- 历史最严重拥堵：2023年春运期间，进港大道、徐海大道、环半岛公路三条主要道路全线拥堵，排队长度约20公里，路上动态滞留车辆约1万辆，持续约5天
- 琼州海峡年停航次数：每年因台风、大雾等恶劣天气停航约10-20次（台风季7-10月尤为频繁），每次停航后恢复通航极易引发集中出行和大规模拥堵

由于港口运力有限（日消化能力约8,000-10,000辆，单船载车约65辆/载客约165人），当车辆到达速率超过港口消化能力时，极易在进港通道形成大规模拥堵。春运期间日均车流量达平日的6倍以上，远超港口消化能力，拥堵问题尤为突出。此外，琼州海峡每年因台风、大雾等恶劣天气停航约10-20次，停航期间车辆积压，恢复通航后集中出行形成"堰塞湖效应"，是除春运外最主要的拥堵诱因。

**拥堵的真实成因（区别于一般理解）：**

预约系统已经从源头控制了"装船车辆"总量——港口根据运力和班次主动放出可预约数量，预约总量本身不会超过港口消化能力。但实际拥堵仍频繁发生，其真实成因是：

- **早到车流叠加**：预约规则原则上只允许提前2小时进入港口区域，但大量车辆为"保险起见"提前更长时间到达，造成道路上车辆密度远超合规通行量。
- **送人车流加压**：单船载客约165人远多于载车约65辆，节假日期间大量纯坐船人员需要去港口，通过出租车、亲友送行（含摩托车）等方式到港，这些"送行车辆"无需预约即上路，进一步加大道路压力。
- **过路车流叠加**：部分非进港、非送行车辆（如本地通勤、过路车辆）也叠加在同一道路网。

### 2.1.2 现有交通管理手段

目前徐闻县针对进港交通拥堵已建立了一套以"预约管控"为核心的管理体系：

**（1）预约制管控**

所有车辆进入徐闻港必须提前通过预约系统预约，未预约车辆不允许进入港口区域。预约平台同时承接车辆预约和人员预约，是控制进港车辆总量的核心手段。

**（2）道路分流与临时停车**

在拥堵发生或逐步加剧时，徐闻县政府动态调整道路功能：
- 将次要道路临时划为停车区域，降低主要道路车辆通行压力；
- 通过车道功能的临时调整，组织已预约进港车辆有序排队，同时保障非进港车辆和过路车辆的正常通行。

**（3）人工设卡检查**

在即将拥堵或拥堵逐步增加时，在关键路口设置检查点，安排人员核验车辆预约状态：
- 未预约车辆不允许继续前行；
- 已预约但**未到预约时段**（早于预约时间 2 小时以上）的车辆，引导其先到附近停车场或临时停车区休息，按时再来。

**（4）信息发布**

通过道路诱导屏、广播等方式发布分流指引、停车区指引等信息。

**（5）视频监控**

主要路段和港口区域已部署视频监控摄像头、电子警察等，用于日常巡查和事件发现。

## 2.2 存在问题

尽管徐闻县已建立了以预约制为核心的交通管控体系，但在实际运行中仍存在以下突出问题：

### 2.2.1 拥堵预判能力不足

当前交通管理以"被动响应"为主——等到拥堵已经发生或明显加剧后才启动应对措施。缺乏基于多源数据的综合预判能力，特别是无法识别"早到车流压力"这一关键先行指标，难以在拥堵形成之前提前介入。

**具体表现：**
- 无法识别提前到达的车辆群体——预约规则虽限定"只允许提前2小时进入港口区域"，但缺少手段实时识别和拦截早到车辆；
- 无法结合气象预报（如台风恢复通航后的集中出行）提前预警；
- 从发现拥堵到启动分流措施存在明显时间差。

### 2.2.2 多源数据未打通

预约数据、实时路况数据、气象数据、视频监控数据分散在不同系统中，缺乏统一的数据汇聚和关联分析平台。管理人员需要在多个系统间切换查看，无法形成对交通态势的全局认知。

**具体表现：**
- 预约系统只能看到预约量，看不到实时路况；
- 路况数据只反映当前状态，无法结合预约数据预判趋势；
- 气象预警与交通管理决策之间缺乏自动化联动。

### 2.2.3 决策依赖人工经验

何时启动道路分流、何时设置检查点、向哪些预约用户发送通知、建议调整到什么时间到达——这些关键决策目前完全依赖管理人员的个人经验判断，缺乏数据支撑和标准化流程。

**具体表现：**
- 不同值班人员对同一态势的判断和响应可能不同；
- 缺乏量化的分级响应标准（如拥堵指数达到多少启动分流）；
- 预约用户通知的时机和内容缺乏精准性。

### 2.2.4 缺少空中视角能力

当拥堵发生时，地面视角受限，难以快速获取拥堵的全貌信息（拥堵长度、滞留车辆数量、拥堵蔓延方向等）。目前尚未配置无人机，无法提供空中巡查、喊话疏导等辅助手段。

### 2.2.5 缺乏复盘与持续优化能力

每次拥堵处置结束后，缺乏系统化的数据记录和效果评估机制。历史处置经验未能沉淀为可量化、可对比的数据资产，无法支撑策略的持续优化。

**具体表现：**
- 无法回答"上次春运分流措施效果如何"；
- 无法对比不同处置策略的缓解时间和效果；
- 领导汇报材料需要手动整理，缺乏自动化报告能力。

### 2.3.2 业务层面

徐闻港作为琼州海峡北岸核心港口，其交通保障能力直接关系到：
- **粤琼两省互联互通**：徐闻港是海南岛与大陆连接的生命线通道；
- **春运保障**：每年春运期间大量车辆集中过海，交通压力极大；
- **应急安全**：台风停航后恢复通航的集中出行、重大事故等突发事件需要快速响应；
- **民生服务**：减少旅客和司机的等待时间，提升出行体验。

建设智慧交通指挥系统，将现有的"被动响应"模式升级为"主动预判+精准调度"模式，是保障琼州海峡通道安全畅通的必要举措。

从治理视角看，徐闻交通问题的特殊性在于：港口运力、船舶班次、停复航窗口等关键变量并不完全由县域道路管理部门直接控制，但车辆排队、道路拥堵、人员滞留和民生保障压力最终会外溢到县域交通体系。因此，本方案的核心不是单纯建设一块交通大屏，而是帮助徐闻县将"港口压力外溢"转化为可感知、可预警、可调度、可复盘的治理能力。

### 2.3.3 技术层面

当前人工智能、大数据分析、物联网等技术已经成熟，具备支撑智慧交通系统建设的条件：
- 规则引擎和专家经验规则可实现基于多源数据的拥堵预判；
- 大屏可视化技术可实现交通态势的直观呈现；
- 无人机技术可提供低成本的空中巡查和应急服务能力；
- 数据融合技术可打通多个信息孤岛，形成统一决策视图。

## 2.4 需求分析

### 2.4.1 总体需求

建设一套AI驱动的智慧交通指挥决策系统，实现：
- **10秒判断态势**：一屏总览交通全局，AI实时评估风险等级；虚拟助手"小语"语音播报态势结论，无需逐项查看；
- **1分钟拿到方案**：AI根据当前阶段自动推荐处置策略；
- **全程看到AI在干活**：从预判到执行到复盘，AI贯穿全流程。

系统以"早到车流压力 + 道路通行余量 + 气象停复航事件"为核心研判变量，围绕琼州海峡进出岛通道形成统一研判口径。通过提前识别车辆集中到达风险、道路拥堵蔓延趋势和气象停复航影响，为领导决策、值班处置和一线执行提供同一套数据依据。

### 2.4.2 功能需求

基于徐闻县交通管理的实际业务场景，系统需要支撑以下核心能力：

**（1）态势感知与一屏总览**

| 需求 | 说明 |
|------|------|
| 实时路况展示 | 在地图上展示主要道路的拥堵状态、车速、拥堵长度 |
| 预约数据可视化 | 展示当前及未来时段的预约量分布，识别即将到来的高峰 |
| 气象信息融合 | 展示气象预警信息，评估对交通的影响 |
| AI态势摘要 | AI自动生成当前交通态势的文字结论和风险等级 |
| 虚拟助手播报 | 虚拟助手"小语"以语音方式播报态势结论、预警信息和每日通报，降低信息获取门槛 |
| 关键指标看板 | 拥堵指数、预约饱和度、港口消化率等核心KPI |

**（2）智能预判与预警**

| 需求 | 说明 |
|------|------|
| 拥堵风险识别 | 在数据充分接入的条件下，基于预约数据、路况趋势和气象信息识别拥堵风险趋势，支撑提前介入；具体预判时长以实际数据接入情况和规则调优结果为准 |
| 分级预警 | 绿色（平稳）→ 黄色（关注）→ 橙色（严重）→ 红色（紧急）四级预警 |
| 主动提醒 | 预警触发时自动弹窗提醒值班人员，附带分析结论和主要影响因素；虚拟助手同步语音播报预警内容 |

**（3）分级响应与策略执行**

系统需支撑徐闻县现有的分阶段递进响应机制：

| 阶段 | 触发条件 | 系统支撑能力 |
|------|---------|-------------|
| 常态管控 | 拥堵指数正常 | 日常监控，预约数据展示 |
| 预警通知 | 预判即将拥堵 | AI建议向预约用户发送通知，调整到达时间；可派无人机确认路况 |
| 道路分流 | 拥堵逐步增加 | AI推荐分流方案，展示分流路线，指令下达；无人机监控+喊话引导 |
| 全面管控 | 完全拥堵 | 无人机全路段巡查统计，设卡点位建议，资源调度 |

**（5）指挥通信协同**

围绕"一屏指挥、一键通信"理念，提供贯穿处置全流程的通信协同能力。当前阶段以系统内置消息和快捷指令为主，远期对接徐闻县应急局规划建设的应急融合通信系统，实现统一调度。

| 需求 | 说明 |
|------|------|
| 一键呼叫 | 在地图或资源面板上点击人员/单位，一键发起语音呼叫 |

| 任务派单 | 以消息形式派发任务，支持文字/图片/位置/语音 |
| 集群对讲 | 一键对讲全员或指定群组（如"全体交警"、"分流组"） |
| 通信记录留痕 | 所有通话和消息自动归档，支撑复盘追溯 |
| 快捷指令 | 预设常用指令（如"启动分流"、"设卡检查"），一键下达 |

| 需求 | 说明 |
|------|------|
| 滞留规模评估 | 基于拥堵长度和车辆数量，估算滞留人员规模和物资需求量 |
| 物资调度建议 | 根据滞留规模，建议调配矿泉水、食品、应急药品等物资数量 |
| 安置点管理 | 在地图上标注临时安置点位置、容量和当前状态 |
| 发放记录 | 记录物资发放时间、地点、数量，支撑事后统计和汇报 |
| 特殊车辆疏散 | 识别冷链车、危化品车等特殊车辆，优先安排疏散或保障 |
| 人员调度 | 展示应急保障人员（志愿者、工作人员）分布和任务分配 |

**（7）复盘分析与报告**

| 需求 | 说明 |
|------|------|
| 历史回溯 | 时间轴回放历史事件处置全过程 |
| 策略评估 | 对比不同处置策略的效果（缓解时间、缓解率） |
| 趋势分析 | 按日/周/月/季度分析交通趋势 |
| 报告导出 | 一键生成Excel/PDF/PPT格式的分析报告 |

### 2.4.3 典型业务场景

**场景一：节假日高峰预警**

节假日进岛车辆集中到达，系统结合未来预约量、历史同期规律、港口小时消化能力和进港道路通行状态，提前识别预约到达曲线是否超过港口消化能力。若未来2小时存在集中到达风险，系统自动生成风险摘要、错峰通知建议、停车区预启用建议和诱导屏发布内容，辅助值班人员提前干预。

**场景二：台风停航后复航疏运**

海事或港口方发布复航信息后，系统结合停航时长、滞留车辆规模、特殊车辆数量、预约恢复情况和天气恢复趋势，预测积压车辆消化时间，建议按冷链车、危化品车、普通车辆等保障顺序分批放行，并联动应急资源保障饮水、食品、停车和现场秩序。

**场景三：进港大道拥堵处置**

当进港大道车速下降、排队长度增加并向县城通道传导时，系统自动识别当前拥堵阶段，推荐设卡点位、分流路线、临时停车区和诱导屏内容。指挥员确认后，系统记录指令下达、人员到位、诱导信息发布和现场反馈，处置结束后形成复盘材料。

# 第三章 总体设计

## 3.1 设计原则

### 3.1.1 预判优先

从传统的"拥堵发生后被动响应"转变为"基于数据提前预判、主动干预"。通过预约数据、路况趋势、气象信息的综合分析，在拥堵形成之前启动预防措施，将问题消灭在萌芽阶段。

### 3.1.2 AI辅助、人工决策

AI负责数据分析、态势研判、策略推荐，人负责最终决策确认和执行指挥。系统定位为"智能参谋"而非"自动驾驶"，确保关键决策始终有人在回路中。

### 3.1.3 平战结合

系统在日常状态下提供态势监控和数据分析能力，在拥堵或应急状态下无缝切换为指挥调度模式。同一套系统、同一套数据，覆盖从日常到应急的全场景。

### 3.1.4 数据驱动

以预约系统数据为核心决策依据，融合路况、气象、视频等多源数据，用数据说话、用数据决策、用数据评估，替代经验驱动的管理模式。

### 3.1.5 务实可行

基于现有基础设施条件进行设计，不依赖尚不具备的能力（如信号灯远程控制）。策略执行通过诱导屏信息发布、人工设卡、无人机辅助等现有或即将配置的手段落地。

## 3.2 设计思路

### 3.2.1 核心理念

本系统以"**10秒判断态势、1分钟拿到方案、全程看到AI在干活**"为核心设计理念：

- **10秒判断态势**：进入系统即可通过AI摘要栏、风险等级色标、关键指标卡片，在10秒内掌握当前交通全局状态；
- **1分钟拿到方案**：AI根据当前拥堵阶段自动推荐对应处置策略，指挥员1分钟内即可获得可执行的方案；
- **全程看到AI在干活**：从预判预警、策略推荐、执行监控到效果评估，AI贯穿处置全流程，用户始终能看到AI的分析过程和结论。

围绕徐闻县的通道型交通治理特点，系统重点解决三件事：一是提前看见未来车流压力，二是判断港口消化能力是否足以承接车辆到达，三是在压力外溢到县域道路前形成可执行的预警、分流和管控建议。

### 3.2.2 业务模式设计

系统设置三大业务模式，覆盖交通管理全场景：

| 模式 | 定位 | 核心场景 |
|------|------|---------|
| 总览监测模式 | 日常值班，一屏看全局 | 实时态势感知、AI预警、风险评估 |
| 指挥处置模式 | 拥堵/应急响应，策略执行 | 分级响应、策略推荐、指令下达、执行监控 |
| 统计分析模式 | 历史复盘，报告生成 | 趋势分析、策略评估、报告导出 |

三大模式之间的数据流转关系：
- 总览监测模式实时数据 → 统计分析模式历史库；
- 指挥处置模式策略执行记录 → 统计分析模式效果评估；
- 统计分析模式优化建议 → 指挥处置模式策略库更新。

### 3.2.3 分阶段递进响应机制

基于徐闻县现有的拥堵处置实践，系统设计四级递进响应机制：

**第一阶段：常态管控（绿色）**
- 系统状态：日常监控，预约数据正常
- 系统动作：实时展示路况和预约量，AI持续评估风险
- 人工动作：无需干预

**第二阶段：预警通知（黄色）**
- 触发条件：检测到早到车流开始累积，或未来时段预约到达量集中超过港口小时消化能力，或气象/停复航事件导致集中出行风险上升
- 系统动作：弹窗预警 + AI建议向"早到车辆预约用户"推送通知（建议先到附近停车场休息）+ 诱导屏发布"当前到港预计时间"等提示信息 + 提示人员预约用户合理安排出行
- 人工动作：确认是否发送通知，准备分流预案
- 通信协同：通过系统消息通道向值班指挥员、交警值班长推送预警信息，必要时发起一对一语音通报
- 无人机：具备飞行条件时可起飞确认实际路况；不具备飞行条件时，由固定视频监控和现场人员反馈补充确认

**第三阶段：道路分流（橙色）**
- 触发条件：早到车流持续增加且经预约系统通知/诱导屏提示后未明显缓解，进港大道或关键通道车速明显下降，拥堵开始向周边道路传导
- 系统动作：AI推荐分流方案（从策略库筛选适用的主路-替代路分流方案）→ 推荐设卡点位 → 划定临时停车区 → 诱导屏发布分流引导内容
- 人工动作：确认分流方案，安排人员设卡检查预约状态和预约时段
- 通信协同：指挥员一键呼叫交警分流组、诱导屏管理员，下达分流指令；通过集群对讲向"分流组"群组同步任务；任务执行情况通过消息回传，全程留痕
- 无人机：监控分流执行效果，统计拥堵车辆数量和距离，喊话引导早到车辆进入停车区

**第四阶段：全面管控（红色）**
- 触发条件：主通道长时间低速或阻断，港口积压持续扩大，现场保障和应急资源需要统一调度
- 系统动作：资源调度建议 + 全员通知 + 滞留规模评估 + 物资需求预估
- 人工动作：全面管控，持续分流+停车区管理；启动应急物资保障（发放盒饭、饮用水等）；必要时疏散驾驶员和特殊车辆
- 通信协同：拉起多方视频会商（交通、公安、应急、港口、海事），实现跨部门统一指挥；指挥员通过大屏直接发起视频连线和语音调度，无需切换设备
- 无人机：全路段巡查，统计拥堵长度和滞留车辆数量，大范围喊话疏导，数据实时回传大屏
- 民生保障：系统根据滞留车辆数量估算物资需求，展示安置点位置和容量，记录物资发放情况

视频监控、无人机、诱导屏等前端感知和执行设备根据现有网络条件接入系统；涉及公安、港口、海事、气象等外部单位的数据，通过安全边界和授权机制进行共享。系统建设应支持分期扩展，先满足核心态势感知与指挥处置，再逐步接入更多无人机和视频智能识别等数据能力。

### 3.4.2 业务架构

**核心业务流程：**

系统的核心业务流程为"感知→研判→决策→执行→反馈"五步闭环：

1. **数据感知**：实时采集预约数据、路况数据、气象数据、视频数据；
2. **AI研判**：融合多源数据，计算风险等级，识别当前拥堵阶段；
3. **策略推荐**：根据当前阶段，AI推荐对应的处置策略和具体动作；
4. **执行落地**：指挥员确认后，通过通讯系统下达指令，一线人员执行；
5. **效果反馈**：实时监控执行效果，偏差超阈值时提示指挥员研判调整。

**角色协作：**

| 角色 | 在系统中的职责 |
|------|--------------|
| AI系统 | 数据分析、风险预判、策略推荐、偏差检测、效果评估 |
| 虚拟助手 | 语音播报态势结论和预警信息、每日/每周通报、降低指挥员信息获取门槛 |
| 指挥员 | 查看AI分析结论、确认执行策略、下达指令、处理偏差 |
| 一线人员 | 接收任务（设卡/分流/无人机操作）、反馈执行进度 |
| 外部系统 | 提供数据（预约/路况/气象）、接收指令（诱导屏/通知推送） |

### 3.4.3 智能研判设计

智能研判能力定位为"数据辅助研判"，重点通过规则计算、经验库匹配、流程留痕和可追溯机制提升指挥决策效率。系统围绕预约压力、港口消化能力、道路运行状态、气象和停复航事件建立可解释的研判规则，为风险预警、阶段识别和策略建议提供依据。

| 能力方向 | 说明 | 作用 |
|---------|------|------|
| 指标计算 | 对早到车流压力、道路通行余量、气象影响等指标进行统一计算 | 形成风险等级和阶段判断依据 |
| 规则研判 | 按照预设阈值和组合条件识别常态、预警、分流、全面管控等阶段 | 支撑预警触发和处置建议 |
| 经验匹配 | 根据历史案例、常用策略和适用条件匹配相似处置方案 | 为指挥员提供参考方案 |
| 处置留痕 | 记录预警触发、方案确认、指令下达、执行反馈和处置结果 | 支撑复盘评估和责任追溯 |

核心能力由五个子模块组成：

**（1）风险预警模块**

输入：车辆预约数据（未来时段进港车辆预约量及预约时段分布）+ 卡口/电子警察过车数据（关键路段实测车流量、车牌识别）+ 实时路况（拥堵指数、车速、车流量）+ 气象信息（台风/大雾预警、停复航状态）+ 道路通行能力（静态配置）+ 人员预约数据（辅助参考，用于评估节假日出行高峰规模）

输出：未来一段时间的拥堵风险提示、风险等级（绿/黄/橙/红）和主要影响因素

核心计算逻辑（指标说明）：

- **早到车流压力**：将道路上的实际车流量与"当前合规在港区车辆数"进行对比。"道路实际车流量"优先采用关键卡口/电子警察的实测过车数据；在卡口数据未接入时，可采用高德车流量估算或现场反馈替代。"合规在港区车辆数"指当前时间 +/- 2 小时窗口内预约时段对应的车辆总和（即按规则可正常进入港口区域的车辆数）。当道路车流量明显大于这一基准时，说明存在大量提前到达的车辆，构成早到车流压力。进阶能力：在卡口车牌识别与预约平台车牌-时段映射打通后，可逐车判定是否早到，实现精准识别。
- **道路通行余量**：通过实时拥堵指数和车速反映主要道路（进港大道、徐海路、环半岛公路）的实际通行能力剩余空间。数据源采用高德路况 API 提供的路段平均车速和拥堵指数（已确认可接入），无需额外部署测速设备。
- **气象影响系数**：根据天气类型和停复航状态加权（台风恢复通航 / 大雾停航 / 大雾复航 / 正常天气分别对应不同系数）。
- **综合风险分**：以"早到车流压力"为核心，结合"道路通行余量"和"气象影响系数"加权计算，映射到四级预警。

**（2）阶段识别模块**

基于实时指标自动判断当前处于四个阶段中的哪一个：

| 阶段 | 判断条件 |
|------|---------|
| 常态（绿色） | 道路车流量与当前合规在港区车辆数基本匹配，主要道路车速正常 |
| 预警（黄色） | 检测到早到车流开始累积（道路车流量超过合规基准），或气象/停复航事件导致风险升高 |
| 分流（橙色） | 早到车流持续增加且经预约系统通知/诱导屏提示后未明显缓解，主要道路车速明显下降，需启动道路分流 |
| 全面管控（红色） | 主通道低速或阻断，车辆滞留持续扩大，需跨部门统一调度并启动应急保障 |

**（3）策略建议模块**

策略库以**分流道路方案**为核心，按"主路拥堵 → 引导至替代路"的模式组织。每条分流策略由主路、替代路、启用条件、车型规则、配套措施和退出条件等要素构成，作为指挥员决策参考；所有策略建议均需由指挥员确认后执行，系统不直接驱动现场动作。

**策略库结构（核心字段）：**

| 字段 | 说明 |
|------|------|
| 主路（A） | 发生拥堵的进港主通道路段（如进港大道关键段、徐海路、环半岛公路） |
| 替代路（B） | 用于承接分流车流的备用道路 |
| 启用条件 | 主路达到的拥堵阈值（如车速持续低于阈值 N 分钟、排队长度超过指定值） |
| 车型规则 | 例如 B 路日常限制大车通行，启动分流时对小车临时放行 |
| 配套措施 | 诱导屏发布内容、人工设卡引导点位、无人机喊话路线、临时停车区位置 |
| 退出条件 | 主路恢复正常通行的判定标准 |

**示例策略（结合徐闻县进港路网梳理，最终以业务方现场踏勘和确认为准）：**

| 编号 | 主路（A） | 替代路（B） | 适用条件简述 |
|------|----------|------------|-------------|
| S-01 | 进港大道关键段 | 进港大道东侧并行替代路 | 主路连续低速时启用，临时放开小车通行，配合诱导屏引导 |
| S-02 | 徐海路 | 环半岛公路 | 主路低速持续时启用，多车道分流，配合诱导屏与设卡引导 |
| S-03 | 进港大道-徐海路全段 | 多路分流 + 临时停车区 | 主路阻断或大范围低速时启用，配合应急资源调度 |
| ... | ... | ... | 由业务方持续梳理扩展 |

策略库为可维护、可扩展的规则集合，由业务方与系统协同梳理、新增、修改和停用。

**按阶段的策略调用方式：**

| 阶段 | 策略调用 |
|------|---------|
| 预警 | 不启用分流策略，主要推送预约用户通知（早到车辆建议先到停车场、人员预约用户合理安排出行）和诱导屏提示 |
| 分流 | 系统从策略库中筛选适用的分流方案推荐给指挥员；指挥员确认后启用配套业务措施（诱导屏发布、设卡引导、临时停车区等） |
| 全面管控 | 多条分流策略并行启用 + 跨部门视频会商和资源调度 + 应急物资保障 + 无人机巡查统计 + 必要时疏散特殊车辆 |

说明：临时停车区、人工设卡、诱导屏发布等是分流策略落地的**业务执行场景**，由业务方根据现场情况配合策略启用，不属于独立策略类型。

**（4）执行跟踪模块**

策略执行过程中，持续监控实际效果与预期的偏差：
- 按业务需要的频率（与高德路况推送频率匹配，约 1 分钟一次）采集实时指标；
- 优先使用卡口/电子警察实测过车量评估主路和替代路的车流变化，量化分流效果（如：主路过车量下降比例、替代路过车量上升比例）；
- 与策略预期效果对比，偏差超过预设阈值时触发预警；
- 生成调整建议供指挥员研判（如：分流效果不佳，建议增加停车区或扩大设卡点位）。

**（5）效果评估模块**

策略执行完成后，自动评估处置效果：
- 计算缓解时间（从执行到恢复正常的时长）；
- 计算缓解率（执行前后拥堵指数变化、卡口实测过车量变化、道路车流量变化）；
- 计算"早到车流引导成效"（在预约平台支持状态反馈的条件下，统计接到通知后改变行程的预约用户比例；进阶能力下可结合卡口车牌识别交叉验证）；
- 生成效果评分，归档到历史数据库；
- 为统计分析模式提供数据支撑。

# 第四章 整合共享与业务协同

## 4.1 系统整合

### 4.1.1 整合目标

本项目不是孤立建设新系统，而是在现有信息化基础上进行整合升级。通过统一的数据接入平台，将分散在各部门、各系统中的交通相关数据汇聚到智慧交通指挥系统中，形成统一的决策视图。

### 4.1.2 与预约系统对接

预约系统是本项目最核心的数据来源，其数据直接决定了拥堵预判的准确性。预约平台同时承接**车辆预约**和**人员预约**两类数据，是识别"早到车流压力"的关键基础。

| 对接能力 | 数据内容 | 业务价值 |
|---------|---------|---------|
| 车辆预约感知 | 获取当前及未来各时段已预约的进港车辆数量及预约时段 | 计算合规在港区车辆数，识别早到车流压力 |
| 人员预约感知 | 获取未来各时段乘船人员预约量分布 | 辅助评估节假日出行高峰规模 |
| 时段分布分析 | 按小时或更细粒度统计预约到达分布（人/车分别统计） | 识别集中到达时段，支撑错峰引导 |
| 车辆类型识别 | 区分小客车、货车、大巴等车辆类型 | 支撑不同车辆群体的通行组织和保障 |
| 用户通知协同 | 向预约用户发送到达时间调整建议；在预约平台支持用户类型区分的条件下，可分别向车辆预约用户（早到提示）和人员预约用户（出行建议）推送差异化通知 | 引导早到车辆错峰到达，从源头削减道路压力 |
| 预约状态同步 | 获取取消、改签等预约变更信息 | 动态修正未来车流预测 |

**目标对接方式：** 通过预约平台开放接口获取人员和车辆预约数据；如平台支持事件推送，可采用消息推送或变更订阅方式同步预约状态。

**关键能力：** 系统需要具备向预约用户推送通知建议的能力，这是"预警通知"阶段的核心执行手段：
- 对**车辆预约用户**：识别其到达时间是否早于预约时段 2 小时，若是则推送"建议先到附近停车场休息"；
- 对**人员预约用户**：在拥堵态势下推送"建议出行时间和方式（避开高峰、选择公共交通等）"，引导减少送行车流。

**前置条件与替代方案：** 预约平台需开放数据查询和用户触达能力；如暂不支持自动推送，可由系统生成通知文案和目标时段，由值班人员通过预约平台后台或人工渠道发布。

### 4.1.3 与高德地图对接

高德地图提供实时路况数据，是拥堵态势感知的基础数据源。

| 对接能力 | 数据内容 | 业务价值 |
|---------|---------|---------|
| 拥堵状态感知 | 主要道路拥堵等级和拥堵指数 | 快速判断进港通道和周边道路运行状态 |
| 道路速度监测 | 各路段平均车速 | 识别低速路段和拥堵扩散趋势 |
| 排队长度研判 | 拥堵路段排队长度 | 辅助判断拥堵规模和分流必要性 |
| 路况趋势分析 | 近一段时间路况变化趋势 | 判断拥堵是否加剧，为预警和处置提供依据 |

**目标对接方式：** 根据地图服务授权能力获取实时路况、道路速度和拥堵等级，更新频率以采购服务和接口能力为准。

**坐标系：** GCJ-02（高德坐标系），系统全局统一使用此坐标系。

### 4.1.4 与气象系统对接

气象数据是拥堵预判的重要辅助因素，特别是台风停航后恢复通航、大雾等场景。

| 对接能力 | 数据内容 | 业务价值 |
|---------|---------|---------|
| 气象预警感知 | 台风、大雾、暴雨等预警信息 | 提前识别恶劣天气对通航和道路交通的影响 |
| 海况陆况监测 | 风力等级、能见度、降雨等关键指标 | 判断停航风险和通行保障压力 |
| 天气趋势研判 | 未来24-72小时天气变化趋势 | 支撑节假日、台风季等场景下的提前部署 |
| 停复航信息同步 | 海事或港口方发布的停航、复航通知 | 识别车辆积压和复航后集中出行风险 |

**目标对接方式：** 对接气象预警、台风路径、风力、能见度等数据；停航/复航信息可通过海事、港口或应急渠道同步。

**关键场景：** 台风停航后恢复通航是最容易引发大规模拥堵的场景。系统需要在收到复航通知后，立即结合预约数据评估即将到来的车流高峰，提前启动预警。

### 4.1.5 与视频监控系统对接

利用现有视频监控设施，为指挥决策提供实时画面支撑。

| 对接能力 | 数据内容 | 业务价值 |
|---------|---------|---------|
| 实时视频查看 | 主要路段和港口区域摄像头画面 | 支撑现场态势确认和指挥联动 |
| 视频回放调取 | 历史视频片段和关键事件画面 | 支撑事件复盘、过程核查和汇报材料整理 |

**目标对接方式：** 通过既有视频平台或流媒体网关接入实时画面，供大屏查看、指挥联动和复盘调用。具体采用国标平台、RTSP、HLS、WebRTC或其他协议，以现有视频平台能力为准。

**关键要求：** 视频接入需考虑专网访问、权限控制、断流重连、延迟控制和历史回放授权。系统不直接替代视频平台，而是在交通场景中聚合关键点位画面。

### 4.1.6 与卡口/电子警察对接（车辆识别数据）

卡口与电子警察是公安交警体系已部署的固定感知设备，能够提供车辆识别和过车统计数据，是计算"早到车流压力"和验证分流效果的关键数据源。

| 对接能力 | 数据内容 | 业务价值 |
|---------|---------|---------|
| 过车量统计 | 各卡口的小时过车数、车型分布 | 实测道路实际车流量，支撑早到车流压力精确计算 |
| 车牌识别 | 车牌号、过车时间、车型、行驶方向 | 与预约平台数据比对，精准识别"早到车辆" |
| 分流执行验证 | 主路、替代路、设卡点的过车量变化 | 实测分流策略执行效果，支撑偏差监控 |

**核心应用场景：**

| 场景 | 数据使用方式 |
|------|------------|
| 早到车流压力计算 | 主要进港道路卡口的实测过车量 vs 预约平台合规在港区车辆数 |
| 早到车辆精准识别（进阶） | 卡口识别的车牌号 与 预约平台车牌-时段映射比对，逐车判定是否早到 |
| 分流效果实测 | 策略启用前后，主路卡口过车量下降比例 + 替代路卡口过车量上升比例 |
| 滞留车辆动态估算 | 关键卡口过车量与港口消化能力差值累计 |

**目标对接方式：** 通过公安交警卡口/电子警察平台获取过车统计和车辆识别数据。具体协议、字段以平台对接接口为准。

**前置条件与替代方案：**
- **数据安全合规**：车牌属于个人信息，需通过授权脱敏机制接入，仅用于交通治理场景；
- **平台开放能力**：需公安交警协调开放卡口数据查询接口；
- **暂未开放时的替代方案**：使用高德车流量估算 + 现场人员目测反馈，精度较低；
- **第一期建议**：仅接入"小时过车量"聚合数据（无需车牌），降低数据安全门槛；第二期再推进车牌比对能力。

**分期接入建议：**

| 阶段 | 接入内容 | 建设目标 |
|------|---------|---------|
| 第一期 | 关键卡口小时过车量（不含车牌） | 早到车流压力从"估算"升级为"实测" |
| 第二期 | 卡口车牌识别 + 预约平台车牌比对 | 逐车精准识别早到车辆，支撑差异化通知 |

### 4.1.7 与无人机系统对接

未来配置的2台无人机将作为拥堵处置各阶段的重要辅助手段。

| 对接能力 | 数据内容 | 业务价值 |
|---------|---------|---------|
| 空中视频回传 | 无人机实时画面接入大屏 | 补充地面视频盲区，辅助掌握拥堵全貌 |
| 任务状态同步 | 无人机起飞、巡查、返航等状态 | 便于指挥中心掌握无人机任务执行情况 |
| 飞行轨迹展示 | 在地图上展示无人机实时位置和巡查路线 | 支撑巡查路线管理和现场态势联动 |
| 车辆识别与统计 | 基于空中视角的车辆数量、排队长度、拥堵范围识别 | 实测排队长度和滞留规模，比地面数据更全面 |

**目标对接方式：** 通过无人机视频流和无人机管理平台获取实时画面、飞行状态、轨迹信息和识别数据。

**使用场景（贯穿多个阶段）：**
- 预警阶段：起飞确认实际路况，为决策提供空中视角依据；
- 分流阶段：监控分流执行效果，统计拥堵车辆数量和距离，喊话引导车辆进入停车区；
- 全面管控阶段：全路段巡查，统计拥堵长度和滞留车辆数量，大范围喊话疏导；
- 物资保障阶段：实测滞留车辆数量，支撑应急物资需求量估算。

**能力边界：** 无人机定位为移动补盲和应急辅助感知能力。受天气、空域、飞手、续航、夜间能见度等条件影响时，系统应支持通过固定视频、卡口数据和现场人员反馈替代确认。

### 4.1.8 与诱导屏/广播系统对接

诱导屏是策略执行的重要信息发布渠道。

| 对接能力 | 数据内容 | 业务价值 |
|---------|---------|---------|
| 诱导信息发布 | 分流指引、停车区指引、安全提示等内容 | 将指挥策略转化为面向驾驶员的现场引导 |
| 发布状态查看 | 当前显示内容、设备在线状态等信息 | 掌握诱导屏运行状态，便于核查发布效果 |

**目标对接方式：** 现有诱导屏已具备远程发布能力，本系统通过诱导屏控制系统或信息发布平台下发分流指引、停车区指引和安全提示。具体协议（如MQTT、HTTP、专有协议等）以现有诱导屏平台对接接口为准。

### 4.1.9 与应急融合通信系统对接（预留）

徐闻县应急局已规划建设应急融合通信项目，本系统在指挥处置场景下预留与融合通信系统的集成能力。

| 对接能力 | 数据内容 | 业务价值 |
|---------|---------|---------|
| 语音调度 | 对讲、电话、集群通信统一调度 | 指挥员可在系统内直接呼叫一线人员，无需切换设备 |
| 视频会商 | 多方视频连线 | 支撑跨部门联合指挥和现场画面共享 |
| 消息通信 | 文字、图片、位置等消息 | 任务派发和执行反馈的统一通道 |
| 通信记录 | 通话和消息记录留痕 | 处置过程可追溯，支撑复盘和责任确认 |

**对接策略：** 应急融合通信系统建成后，本系统通过标准接口集成其通信能力，在指挥处置模式中嵌入语音调度、视频会商和消息通信入口，实现"一屏指挥、一键通信"。在融合通信系统建成前，系统通过内置消息和快捷指令提供基础协同能力。

### 4.2.2 数据标准规范

为确保多源数据的有效融合，制定以下数据标准：

**（1）坐标系统一**
- 全系统统一使用 GCJ-02 坐标系（高德坐标系）；
- 外部数据如使用 WGS-84 坐标系，接入时自动转换。

**（2）时间格式统一**
- 统一使用 ISO 8601 格式：`YYYY-MM-DDTHH:mm:ssZ`；
- 时区统一为东八区（UTC+8）。

**（3）接口规范统一**
- RESTful API 遵循 OpenAPI 3.0 规范；
- 请求/响应统一使用 JSON 格式；
- 统一错误码体系（业务错误码 + HTTP状态码）。

**（4）数据质量要求**
- 数据完整性：关键字段不允许为空；
- 数据时效性：超过有效期的数据标记为过期；
- 数据异常检测：接入层自动检测数据断流和异常值。

### 4.2.3 数据安全与权限

| 安全措施 | 说明 |
|---------|------|

## 4.3 业务协同

### 4.3.1 跨部门协同机制

智慧交通指挥系统涉及多个部门的协同配合：

| 部门 | 协同职责 | 系统支撑 |
|------|---------|---------|
| 交通局 | 道路分流决策、诱导屏信息发布 | 分流方案推荐、诱导屏内容推送 |
| 公安局/交警 | 设卡检查、交通管制 | 设卡点位建议、预约核验信息 |
| 应急管理局 | 台风/极端天气应急联动、物资保障、人员疏散 | 气象预警联动、应急预案触发、物资调度建议、安置点管理 |
| 无人机飞手 | 无人机起飞、巡查、喊话 | 任务派发、巡查路线建议 |
| 预约平台运营方 | 用户通知推送 | 通知内容生成、推送指令 |

### 4.3.2 分阶段协同流程

**阶段一：常态管控**
- 协同方：无（系统自动监控）
- 系统动作：持续采集数据，AI评估风险

**阶段二：预警通知**
- 协同方：预约平台运营方 + 无人机飞手（视情况）
- 系统动作：生成通知内容 → 推送至预约平台 → 平台向用户发送
- 通信方式：系统消息推送给值班指挥员和交警值班长；必要时通过一对一语音通报
- 无人机动作：可起飞确认实际路况，为是否启动分流提供空中视角依据
- 协同要点：通知内容需包含建议到达时间、当前路况、替代方案

**阶段三：道路分流**
- 协同方：交通局 + 公安局/交警 + 无人机飞手
- 系统动作：推荐分流方案 → 诱导屏信息推送 → 设卡点位建议
- 通信方式：指挥员通过系统一键呼叫交警分流组、诱导屏管理员；集群对讲下达统一指令；任务执行通过消息和图片回传留痕
- 无人机动作：监控分流执行效果，统计拥堵车辆数量和距离，喊话引导车辆进入停车区
- 协同要点：
  - 交通局确认分流路线和停车区划定；
  - 公安局安排人员到检查点核验预约；
  - 确保非进港车辆和出租车通行不受影响；
  - 无人机统计数据实时回传大屏，辅助指挥员评估分流效果。

**阶段四：全面管控**
- 协同方：全部门 + 无人机飞手
- 系统动作：全员通知 → 资源调度
- 通信方式：在大屏上拉起多方视频会商（交通/公安/应急/港口/海事），实现跨部门统一指挥；语音调度、视频会商、消息派单全部在系统内完成，所有通信记录自动归档
- 无人机动作：全路段巡查，统计拥堵长度和滞留车辆数量，大范围喊话疏导
- 协同要点：
  - 无人机飞手按系统建议路线起飞巡查；
  - 无人机统计数据（拥堵长度、车辆数）回传系统；
  - 喊话内容由指挥员通过系统确认后执行。

### 4.3.3 通讯协同机制

系统内置基础通讯能力，支撑跨部门实时协同；待应急局融合通信系统建成后，可集成其通信能力实现统一调度：

**当前阶段（系统内置）：**

| 通讯方式 | 适用场景 |
|---------|---------|
| 系统消息 | 任务派发、进度反馈、状态通知 |
| 快捷指令 | 预设常用指令（如"启动分流"、"设卡检查"），一键下达 |
| 视频连线 | 与现场人员/无人机画面实时连线 |

**远期集成（对接应急融合通信系统后）：**

| 通讯方式 | 适用场景 |
|---------|---------|
| 融合语音调度 | 指挥员在大屏上一键呼叫一线人员、交警、飞手等 |
| 多方视频会商 | 跨部门联合指挥，多方画面同屏 |
| 统一消息通道 | 文字/图片/位置消息统一收发，替代多个群聊 |
| 通信记录留痕 | 所有通话和消息自动归档，支撑复盘追溯 |

## 4.4 数据接口

### 4.4.1 核心数据对接能力说明

本方案不预设外部系统的具体接口路径和字段格式，具体协议、字段、频率和权限以实际系统对接确认为准。解决方案阶段重点明确需要哪些数据能力、用于哪些业务场景，以及在接口暂未具备时的替代路径。

| 能力名称 | 数据方向 | 核心数据项 | 业务用途 | 前置条件/替代方案 |
|---------|---------|-----------|---------|----------------|
| 车辆预约数据查询 | 预约平台 → 智慧交通系统 | 进港车辆预约量、预约时段分布、车辆类型、预约状态 | 计算合规在港区车辆数，识别早到车流压力 | 预约平台开放数据接口；暂未开放时可采用定时数据交换或人工导入 |
| 人员预约数据查询 | 预约平台 → 智慧交通系统 | 乘船人员预约量、时段分布、预约状态 | 辅助评估节假日出行高峰规模 | 预约平台开放数据接口；暂未开放时可采用定时数据交换或人工导入 |
| 用户通知建议 | 智慧交通系统 → 预约平台/运营人员 | 目标时段、建议到达时间、通知内容（区分车辆/人员预约） | 对早到车辆引导先到停车场休息；对人员预约用户提供出行建议 | 平台支持推送时自动触达；暂不支持时输出通知文案供人工发布 |
| 港口通行数据 | 港口/海事 → 智慧交通系统 | 班次、待渡车辆、停复航状态、预计消化时间 | 判断港口消化能力和复航疏运压力 | 港口或海事渠道开放数据；暂未开放时采用人工填报或定时报送 |
| 路况态势数据 | 地图服务/交管数据 → 智慧交通系统 | 道路车速、拥堵等级、拥堵长度、趋势变化 | 识别道路拥堵和扩散趋势 | 地图路况服务授权；暂未接入时采用视频、卡口和现场反馈替代 |
| 卡口过车数据 | 公安交警卡口/电子警察平台 → 智慧交通系统 | 关键卡口小时过车量、车型分布、车速 | 实测道路车流量，支撑早到车流压力精确计算和分流效果验证 | 第一期聚合数据接入；第二期推进车牌识别与预约数据比对，需公安交警协调和数据安全合规授权 |
| 气象预警数据 | 气象/应急渠道 → 智慧交通系统 | 台风、大雾、暴雨、风力、能见度 | 判断停航风险和恶劣天气影响 | 接入气象预警渠道；必要时支持人工录入关键预警 |
| 视频监控接入 | 视频平台 → 智慧交通系统 | 关键点位实时画面、必要的历史回放 | 现场确认、指挥联动和处置复盘 | 视频平台授权和专网连通；暂未接入点位通过现场反馈补充 |
| 无人机辅助感知 | 无人机平台 → 智慧交通系统 | 实时画面、飞行状态、轨迹、车辆识别统计 | 移动补盲、巡查确认、排队长度实测、喊话辅助 | 无人机设备、飞手和飞行条件满足；不满足时使用固定视频、卡口数据和现场反馈 |
| 诱导信息发布 | 智慧交通系统 → 诱导屏/发布平台 | 分流提示、停车区指引、安全提示 | 执行交通诱导和分流组织 | 诱导屏平台支持远程发布；暂不支持时输出发布内容供人工下发 |

### 4.4.2 分期接入建议
