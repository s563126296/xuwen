# 指挥模式 v2.0 — 公共上下文

> **所属模块**：指挥模式（Command）
> **使用端**：PC 大屏
> **前置版本**：v1.0-demo-2026-04-24

---

## §1 产品定位变化

### v1.0 定位
指挥模式是"拥堵处置决策中枢"，当拥堵指数 > 4 时触发，提供：
- 拥堵归因分析
- AI 策略推荐
- 执行跟踪
- 效果预测

### v2.0 定位
指挥模式升级为"日常拥堵 + 应急场景的统一指挥中枢"，覆盖：
- 日常拥堵处置（原有能力）
- 应急场景指挥（从应急模式迁入）
- 根据场景类型动态调整面板

一句话定义：

> 指挥模式 v2.0 = "日常 + 应急"的统一指挥入口，场景感知，能力分层。

---

## §2 为什么要合并

### 业务背景
1. 应急业务过于复杂，实际操作中系统难以完全覆盖
2. 业务上强调"指挥"，轻"应急"
3. 5 个模式对用户来说切换成本高，容易混淆

### 合并价值
1. 降低模式数量：5 个 → 4 个
2. 统一指挥入口，减少切换
3. 保留应急核心能力，去掉过度复杂的功能
4. 指挥模式成为"遇事就来这里"的统一决策中心

---

## §3 合并方案

### 方案选择
采用"指挥模式吸收应急核心功能"方案，而非创建全新模式。

### 保留项（指挥模式原有）
- 6 步决策链
- 拥堵归因分析
- AI 策略推荐
- 策略冲突检测
- 执行资源面板
- 历史策略统计
- 视频通信系统
- 地图视频联动

### 迁入项（从应急模式）
- 特殊车辆追踪（冷链车/危化品车）
- 物资需求估算（简化版）
- 跨部门任务板（整合到执行资源面板）
- 应急预案快捷入口

### 移除项（应急模式不迁入）
- 5 阶段状态机（预警期/停航初期/滞留高峰期/复航准备期/复航消化期）
- 滞留预测模型
- 台风路径动画
- 视频会商系统（保留基础通信）
- 底部滞留趋势时间轴
- 应急模拟器

## §4 场景感知机制

合并后的指挥模式不是简单堆砌功能，而是根据场景动态调整。

### 场景类型
```typescript
type CommandScene = 'congestion' | 'emergency';
```

### 日常拥堵场景（congestion）
- 触发条件：拥堵指数 > 4
- 摘要条：橙色，显示拥堵态势
- 策略面板：显示 AI 策略推荐
- 执行资源：显示人员/物资
- 地图：拥堵热力 + 分流路线

### 应急场景（emergency）
- 触发条件：港口停航 / 台风预警
- 摘要条：红色，显示预警信息 + 滞留数
- 策略面板：顶部增加"预案快捷入口"
- 执行资源：增加特殊车辆追踪 + 物资需求
- 地图：叠加停车区 + 无人机巡查路线
- 通信面板：自动切换到应急频道

## §5 布局变化

### v1.0 布局
```
┌─────────────────────────────────────┐
│ 摘要条（拥堵态势）                   │
├──────────────┬──────────────────────┤
│              │                      │
│   地图       │   策略面板（右侧）    │
│              │                      │
├──────────────┴──────────────────────┤
│ 底部通信面板                         │
└─────────────────────────────────────┘
```

### v2.0 布局
```
┌─────────────────────────────────────┐
│ 摘要条（场景感知：拥堵/应急）         │
├──────────────┬──────────────────────┤
│              │  策略面板（右侧）     │
│              │  · AI 策略推荐       │
│   地图       │  · 预案入口（应急时） │
│              │  · 历史统计          │
│              ├──────────────────────┤
│              │  执行资源面板（右侧） │
│              │  · 人员/物资/设备    │
│              │  · 任务板（应急时）   │
│              │  · 特殊车辆（应急时） │
│              │  · 物资需求（应急时） │
├──────────────┴──────────────────────┤
│ 底部通信面板（频道切换）              │
└─────────────────────────────────────┘
```

说明：
- 右侧拆成上下两个面板
- 上：策略面板（保留）
- 下：执行资源面板（扩展）
- 应急场景时，执行资源面板增加折叠卡片

## §6 目标用户与新增价值

| 角色 | v1.0 使用方式 | v2.0 新增价值 |
|------|-------------|--------------|
| 指挥员 | 处置日常拥堵 | 同一入口处置应急场景，不用切模式 |
| 县领导 | 查看策略执行 | 应急场景也能看到资源调度和任务进度 |
| 跨部门 | 接收策略指令 | 应急时可看到任务分配和物资需求 |

---

## §7 研发原则

1. **保留指挥模式核心架构**
2. **应急功能作为"可选面板"叠加**
3. **场景切换不改变主体布局**
4. **预案简化为快捷入口，不做复杂状态机**
5. **移除应急模式代码前先隐藏，降低风险**

---

## §8 优先级

### P0（必须先做）
- commandStore 扩展（增加 scene、specialVehicles、supplyDemand、emergencyTasks）
- 场景切换逻辑
- 3 个核心组件迁移（特殊车辆、物资需求、任务板）
- 摘要条场景切换

### P1（第二阶段）
- AI 归因分析增强（多因素权重）
- 策略推荐优化（成功率显示）
- 效果预测可视化（置信区间）

---

## §9 风险与挑战

### 技术风险
- 功能整合可能导致界面过于复杂
- 场景切换逻辑需要清晰的状态管理

### 缓解措施
- 保持指挥模式简洁性
- 应急功能作为"可折叠面板"
- 场景切换只改变面板内容，不改变布局

### 业务风险
- 简化后可能无法满足真实应急场景需求

### 缓解措施
- 与项目经理确认保留的核心功能清单
- 先隐藏应急模式，不直接删除代码
