## 一、项目背景与愿景

### 1.1 项目背景

徐闻县位于中国大陆最南端，隔琼州海峡与海南岛相望，是海南自贸港封关运作后**唯一的陆路口岸**。2025年12月18日海南正式启动全岛封关运作，实行"一线放开、二线管住"，琼州海峡成为"二线"的核心物理边界。

**核心数据（2024年）**：琼州海峡年运输旅客**1914万人次**、车辆**552万辆次**、航班**8.1万班次**；常态运营**54艘**船舶、**8个**港口；日最高旅客量18.8万人次、日最高车辆量3.9万辆次。封关后首个春运（2026年1-2月）过海旅客464.4万人次，封关百日外贸进出口同比增长32.9%。

徐闻的交通枢纽地位正发生质的飞跃——从普通的县级交通节点升级为**国家级口岸交通枢纽**。

**核心矛盾**：当前智慧交通大屏仅实现了基础数据展示功能，与徐闻"海南门户"的战略定位严重不匹配：
- 功能层面：仅有车流统计和简单拥堵展示，缺乏港口运营、海峡通行、口岸通关等核心场景
- 数据层面：依赖海康卡口单一数据源，港口数据为静态预测值，拥堵数据源已失效
- 智能层面：拥堵预测模型局限（仅2卡口+30分钟窗口），调度策略为静态图文
- 视觉层面：标准深蓝看板模板，缺乏3D场景和视觉冲击力

### 1.2 客户愿景

> "徐闻毗邻海南，在海南封关的前提下，徐闻是海南唯一的陆路口岸，港口交通和运输至关重要，智慧交通驾驶舱要做的更加智能化和高级感"

### 1.3 目标定位

| 维度 | 现状 | 目标 |
|------|------|------|
| **产品定位** | 交通数据看板 | **港城融合智慧交通决策中枢** |
| **核心叙事** | "徐闻有多少车" | **"海南门户，通达天下"** |
| **用户价值** | 看数据 | **用数据做决策** |
| **视觉标杆** | 普通大屏模板 | **数字孪生电影级体验** |

## 二、现有系统诊断

### 2.1 原始材料分析摘要

已分析5份原始材料：

| 文件 | 核心发现 |
|------|---------|
| **交通大屏设计图.jpeg** | 标准深蓝底+左中右三区布局，中央为2D地图+路况叠加，左侧港口拥堵分析，右侧交通流量；视觉层次单一，无3D效果 |
| **驾驶舱交通主题业务指标.xlsx** | 40个功能项，2大一级模块（交通态势分析+交通态势地图），6个二级模块；功能以简单统计为主，缺乏深度分析 |
| **徐闻交通大屏数据来源和数据指标.xlsx** | 数据源主要为海康卡口（1h更新）、港口提供的一天静态数据；设备点位约120个；重点路段通过两端卡口差值计算 |
| **智慧交通大屏原型优化功能说明.docx** | 最近一次优化（7月24日）主要调整港口拥堵展示（进港/出港方向）、拥堵预测（5分钟更新/30分钟窗口）、天气展示、卡口弹窗优化 |
| **徐闻县交通拥堵模型分析报告.pdf** | 拥堵预测模型从初版（288条+2卡口+XGBoost MSE=0.33）优化到（17520条+多卡口+XGBoost/CatBoost），但仍有4大局限 |

### 2.2 问题清单（6大类18项）

#### 问题类型一：数据维度单一（4项）

| # | 问题 | 影响 | 严重度 |
|---|------|------|--------|
| D1 | 卡口数据仅1小时粒度 | 无法分钟级感知突发拥堵 | 高 |
| D2 | 港口运力是静态预测数据（一天排班表循环播放） | 港口状态完全失真 | 严重 |
| D3 | 拥堵数据依赖百度API（**已被封禁**） | 拥堵指数核心数据缺失 | 严重 |
| D4 | 缺乏多源数据融合 | 无AIS、ETC、浮动车数据 | 高 |

#### 问题类型二：指标体系浅薄（3项）

| # | 问题 | 影响 | 严重度 |
|---|------|------|--------|
| I1 | 40个功能以简单计数为主（今日车流量、7日趋势） | 无业务洞察价值 | 高 |
| I2 | 只有"哪里堵"，没有"为什么堵" | 无法支撑决策 | 高 |
| I3 | 缺乏跨域关联分析 | 港口-路网-气象无法联动 | 高 |

#### 问题类型三：港口特色缺失（4项）

| # | 问题 | 影响 | 严重度 |
|---|------|------|--------|
| P1 | 无船舶实时位置和航行状态 | "海南门户"无从体现 | 严重 |
| P2 | 无港口通关效率指标 | 封关后核心场景缺失 | 严重 |
| P3 | 无琼州海峡整体交通态势 | 海峡通行能力不可见 | 高 |
| P4 | 无两港对比分析 | 无法统筹调度 | 中 |

#### 问题类型四：交互深度不足（3项）

| # | 问题 | 影响 | 严重度 |
|---|------|------|--------|
| X1 | 无下钻路径（总览→路段→卡口→视频） | 无法深入排查 | 高 |
| X2 | 地图仅点位标注+红色叠加，无图层联动 | 地图利用率低 | 高 |
| X3 | 无时间回溯、多维筛选 | 无法历史分析 | 中 |

#### 问题类型五：智能化程度低（2项）

| # | 问题 | 影响 | 严重度 |
|---|------|------|--------|
| A1 | 拥堵预测仅30分钟窗口，模型局限性大 | 预测价值有限 | 高 |
| A2 | 调度策略是静态图文 | 无法适应实时变化 | 高 |

#### 问题类型六：视觉缺乏冲击力（2项）

| # | 问题 | 影响 | 严重度 |
|---|------|------|--------|
| V1 | 标准深蓝看板模板，无3D效果 | 视觉档次不够 | 高 |
| V2 | 无粒子流动、光效动画 | 缺乏科技感和生命力 | 中 |

## 五、颠覆式创新方案

### 5.1 创新一：数字孪生港口

**现状**：港口数据为静态排班表循环播放
**创新**：构建港口完整3D数字孪生

```
创新内容：
├── 3D港口模型（泊位+候船厅+查验通道+停车区）
├── 实时船舶3D模型（位置+装载+航速）
├── 车辆流动粒子（进出港通道实时车流）
├── 港区热力图（车辆/旅客密度分布）
└── 环境模拟（天气+光影+波浪）
```

**依赖新数据**：港口闸机实时数据、AIS船舶数据

### 5.2 创新二：海峡通行指数

**现状**：无海峡整体态势视图
**创新**：首创"海峡通行指数"综合评估模型

```
海峡通行指数 = f(船舶密度, 平均航速, 风力, 浪高, 能见度, 通航通知)

0-40:  🟢 通畅（正常通航）
40-60: 🟡 一般（建议关注）
60-80: 🟠 紧张（建议限流）
80-100: 🔴 严峻（建议停航）
```

**依赖新数据**：AIS系统、海洋气象

### 5.3 创新三：港城一体联动分析

**现状**：港口和城市交通完全割裂
**创新**：建立"港口状态↔城市路网"的因果链路分析

```
示例链路：
大船到港（H-02）
  → 预测30分钟后出港车辆高峰 200辆（A-02）
  → 出港通道拥堵预测升至7.5（A-01）
  → 建议提前调整信号灯配时（A-10）
  → 诱导屏发布分流信息（E-07）
```

### 5.4 创新四：AI交通智脑

**现状**：拥堵预测30分钟+静态调度策略
**创新**：大模型驱动的交通智能体

| 能力 | 描述 |
|------|------|
| **智能问答** | "为什么港口路这么堵？" → AI归因分析+可视化 |
| **预测增强** | 2小时拥堵预测+4小时港口压力预测 |
| **智能调度** | 实时生成调度策略替代静态图文 |
| **自动报告** | AI生成每日/每周交通运行报告 |
| **异常解读** | 自动解释异常指标波动原因 |

### 5.5 创新五：应急模式一键切换

**现状**：无应急场景支持
**创新**：台风/停航/事故等场景一键切换应急布局

```
正常模式                          应急模式（台风停航）
┌──────────┐                    ┌──────────────────┐
│ 日常三栏  │     一键切换 →     │ 🔴 台风预警       │
│ 布局      │                    │ 滞留预测 | 分流方案│
│          │                    │ 应急资源 | 视频墙  │
└──────────┘                    └──────────────────┘
```

### 5.6 创新六：时间机器

**现状**：只能看实时数据
**创新**：

- **历史回溯**：拖动时间轴回放任意时段全域态势
- **模拟推演**：输入台风/春运/施工等参数，预测交通影响
- **场景对比**：工作日vs周末、今年vs去年同期对比分析

## 六、数据需求与可行性

### 6.1 数据接入优先级

#### P0（必须接入，决定系统成败）

| 数据源 | 数据内容 | 可行性评估 | 接入方式 |
|--------|---------|-----------|---------|
| **港口闸机系统** | 实时进出港车辆、候船排队 | ★★★★☆ 港口方有系统，需协调API | 实时API |
| **AIS船舶系统** | 船舶位置、航速、航向 | ★★★★★ 公开AIS数据+船讯网API | API/WebSocket |
| **高德路况API** | 替代已失效的百度API | ★★★★★ 高德路况API成熟稳定 | REST API |

#### P1（重要增强，显著提升体验）

| 数据源 | 数据内容 | 可行性评估 | 接入方式 |
|--------|---------|-----------|---------|
| **海康卡口升级** | 5分钟级过车数据 | ★★★★★ 海康支持更高频率 | 配置调整 |
| **视频AI分析** | 异常检测、排队估算 | ★★★★☆ 海康/华为AI分析能力成熟 | AI平台API |
| **港口调度系统** | 船舶排班、装载实时数据 | ★★★☆☆ 需港口方深度配合 | API对接 |
| **海洋气象** | 风力/浪高/能见度/台风 | ★★★★★ 气象局公开接口 | API |

#### P2（锦上添花，封关后必须）

| 数据源 | 数据内容 | 可行性评估 | 接入方式 |
|--------|---------|-----------|---------|
| **海关系统** | 通关时长、查验数据 | ★★☆☆☆ 海关系统封闭，需上级协调 | 待评估 |
| **ETC数据** | 高速出入口流量 | ★★★☆☆ 需交通运输局协调 | API |
| **海事局** | 通航通知、航道管控 | ★★★☆☆ 需海事局协调 | API/推送 |

#### P3（远期目标）

| 数据源 | 数据内容 | 可行性评估 |
|--------|---------|-----------|
| 交通诱导屏控制 | 诱导屏内容远程更新 | 需交管局配合 |
| 应急指挥系统 | 应急资源和预案联动 | 需应急办配合 |
| 信号灯控制 | 信号灯配时远程调整 | 需交管局配合 |

### 6.2 数据差距分析

```
当前数据覆盖率：约 35%（39/110 指标可直接/间接获取）

P0 接入后覆盖率：约 60%（66/110）
P0+P1 接入后覆盖率：约 80%（88/110）
全部接入后覆盖率：约 95%（105/110）

无法通过数据接入解决的指标（约5项）：
- 模拟推演类指标（需模型开发）
- 车辆行为画像（需算法开发）
- AI调度建议（需大模型定制）
```
