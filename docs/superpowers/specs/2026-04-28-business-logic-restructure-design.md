# 徐闻智慧交通大屏 — 业务逻辑重构设计文档

> 版本：V1.0 | 日期：2026-04-28
> 状态：已确认 | 优先级：P0

---

## 一、设计目标

基于 v2.0 需求分析（0427 讨论整理），从用户故事出发重新梳理三个模式的完整业务闭环，解决当前系统的 4 个核心断点：

1. 策略和资源没有形成因果关系
2. 偏差没有分清"执行问题"还是"环境变化"
3. AI 没有把"修正判断"讲清楚
4. 学习没有沉淀成可展示的业务资产

---

## 二、三模式 AI 角色定位

| 模式 | AI 角色 | 核心价值 | 形象 |
|------|---------|---------|------|
| 总览模式 | AI 值班助手 | 主动感知、主动播报、随时可问 | 小语（虚拟人形象） |
| 指挥模式 | AI 参谋长 | 给策略、盯执行、发现偏差、主动纠偏 | 无虚拟人，面板/弹窗呈现 |
| AI 分析模式 | AI 策略实验室 | 推演未来、复盘过去、展示进化 | 无虚拟人，图表/记录呈现 |

---

## 三、跨模式数据流

```
                    分级触发                    复盘数据
┌──────────┐  ──────────────→  ┌──────────┐  ──────────→  ┌──────────┐
│  总览模式  │                   │  指挥模式  │               │ AI分析模式 │
│ AI值班助手 │  ←──────────────  │ AI参谋长  │  ←──────────  │AI策略实验室│
└──────────┘    退出返回         └──────────┘   反哺推荐     └──────────┘
```

### 分级触发（总览→指挥）

| 风险等级 | 条件 | 系统行为 |
|---------|------|---------|
| 低风险 | AI 预测拥堵指数将达 4-5 | 总览模式内提醒，不建议切换 |
| 高风险 | AI 预测拥堵指数将达 >6 | 建议提前进入指挥模式，AI 已预先准备归因+策略 |
| 紧急 | 事故/停航/拥堵指数实际 >6 | 自动切换指挥模式 |

### 复盘数据（指挥→AI分析）

每次策略执行结束后自动写入：
- 策略版本迭代记录（v1→v2→...）
- 每个检查点的预期 vs 实际值
- 偏差事件及归因（策略问题 / 执行问题 / 环境变化）
- AI 新学到的因子
- 用户评价和补充说明

### 反哺推荐（AI分析→指挥）

策略推荐卡片上明确展示：
- "基于 N 次历史执行 | 当前条件匹配度 X% | 置信度 Y%"
- 历史成功率
- 当前条件下的预期效果（基于历史数据校准）

## 四、核心用户模型

### 使用场景

- 领导和指挥员大概率同时在现场看大屏
- 大屏是决策现场，不是审批系统
- AI 决策需要领导拍板时，领导现场决定即可
- 不需要系统内审批流程

### 角色分工

| 角色 | 职责 | 系统交互 |
|------|------|---------|
| 领导 | 决策拍板、风险把控、效果评价 | 看大屏、现场决定 |
| 指挥员 | 具体操作、资源调度、现场协调 | 操作系统、确认执行 |
| AI 参谋长 | 分析判断、给建议、盯执行、发现偏差 | 先分析→再确认→不越权 |

## 五、指挥模式完整闭环（8 步）

### 第 ① 步：发现拥堵 / 进入指挥模式

**触发方式：** 由总览模式分级触发（低/高/紧急）或手动导航栏进入

**进入时 AI 自动完成：**
- 读取当前 11 个因素的实时数据
- 运行归因引擎，预判拥堵主因
- 预生成策略推荐（指挥员进来就能看到建议）

### 第 ② 步：AI 判因（11 个因素）

**归因引擎输入：**

| 序号 | 因素 | 数据来源 | 改动状态 |
|------|------|---------|---------|
| 1 | 当前拥堵指数 | 实时数据 | 已有 |
| 2 | 道路容量（S376等） | 静态配置 | 已有 |
| 3 | 当前天气 | 气象数据 / Mock | 已有 |
| 4 | 当前时段 | 系统时间 | 已有 |
| 5 | 港口积压情况 | 港口数据 / Mock | 已有 |
| 6 | 历史同类策略效果 | AI 分析模式反哺 | **新增** |
| 7 | 执行资源到位情况 | 资源面板 | **新增** |
| 8 | 分流道路当前路况 | 实时数据 / Mock | 已有 |
| 9 | 大货车占比 | 卡口数据 / Mock | **新增** |
| 10 | 汇入车流 | 卡口数据 / Mock | **新增** |
| 11 | 信号灯配时 | 信号灯系统 / Mock | **新增** |

**归因输出：**
- 主因类型 + 权重排序（如：港口积压 60% + 流量高峰 30% + 天气 10%）
- 归因置信度

### 第 ③ 步：AI 推荐策略

**策略推荐卡片必须包含：**

1. **推荐理由**：为什么推荐这条策略（基于归因结果）
2. **历史依据**：基于 N 次历史执行 | 条件匹配度 X% | 历史成功率 Y% | 置信度 Z%
3. **预期效果**：
   - 资源到位时间（AI 基于历史给出，指挥员可手动修正）
   - 策略生效时间（从资源到位开始算）
   - 总计时间 = 资源到位时间 + 策略生效时间
   - 各检查点预期值（30/60/90/120 分钟）
4. **所需资源**：
   - 人员：需要几人 / 当前在岗几人 / 预计到达时间
   - 物资：需要多少 / 库存多少
   - 设备：需要几台 / 在线几台
   - 是否满足执行条件（✓/✗）

**执行资源与策略绑定：** 不同策略显示不同的资源需求，AI 给出推荐配置，指挥员可快捷调整。

### 第 ④ 步：指挥员确认资源

**资源面板从"展示"升级为"决策依据"：**
- 显示每个资源项的满足状态（✓/✗）
- 人员显示预计到达时间（基于历史平均）
- 指挥员可手动修正"资源到位时间"
- 显示总体判断："资源全部满足，可立即执行" 或 "⚠️ 交警不足，建议增派"
- 领导现场拍板，不需要系统审批

### 第 ⑤ 步：执行监控

**AI 后台三通道并行监控：**

| 通道 | 内容 | 优先级 |
|------|------|--------|
| 数据监控 | 实际曲线 vs 预期曲线，偏差实时计算 | P0 |
| 现场反馈 | 聊天窗口文字/图片/短视频，AI 提取关键信息 | P0 |
| 视频监控 | 视频/无人机画面，AI 车流计数/排队检测 | P1 |

**数据监控：**
- 预期曲线（虚线）vs 实际曲线（实线）
- 偏差颜色：绿 ≤5% / 黄 5-10% / 橙 10-15% / 红 >15%
- 时间轴起点 = 资源到位时间（不是点击执行时间）

**现场反馈 AI 提取（P0）：**
- 现场人员通过聊天窗口发文字/图片/短视频
- AI 自动分析，提取关键信息
- AI 行为模式：分析 → 确认 → 询问是否调整（不越权直接行动）

**视频/无人机画面：**
- 拥堵路段实时视频
- 无人机俯拍
- 策略执行现场画面
- 领导可直观看到现场情况

### 第 ⑥ 步：偏差处理

**偏差归因三大类：**

| 类别 | 具体原因 | AI 建议 |
|------|---------|---------|
| 策略问题 | 策略本身不适合当前场景 | 替换策略 |
| 执行问题 | 资源未到位 / 现场障碍 / 执行延迟 | 补充资源 / 解决障碍 |
| 环境变化 | 天气突变 / 新事故 / 车流超预期 / 港口变化 | 调整预期 / 追加策略 |

**AI 行为：**
1. 分析归因 → 确定属于哪一类
2. 给出建议（先分析，再确认，不越权）
3. 指挥员决定：继续 / 调整 / 替换

**策略调整两种处理：**
- **补充性质**（解决执行障碍）→ 合并为版本迭代（v1→v2），更新预期曲线
- **追加性质**（独立新策略）→ 独立监控，各自有预期曲线

### 第 ⑦ 步：策略结束 + 结构化复盘

**触发条件：** 拥堵指数降到目标值以下并持续 15 分钟

**复盘报告内容：**
- 执行概况：策略名称、版本、总耗时、最终效果
- 版本迭代记录：每个版本的内容、原因、预期
- 偏差事件：时间、类型（策略/执行/环境）、原因、处理方式、解决耗时
- AI 学习：新增因子、影响范围、准确率变化
- 用户评价：有效/一般/无效 + 补充说明

### 第 ⑧ 步：AI 学习沉淀

**学习数据自动写入 AI 分析模式：**
- 新发现的因子 → 进化记录
- 预期 vs 实际数据 → 校准模拟器
- 用户评价 → 调整策略推荐权重
- 偏差归因统计 → 决策树更新

## 六、AI 分析模式（4 个功能）

### 6.1 策略模拟器

**用途：** 选择策略 + 调参数 → 实时看模拟效果

**参数面板（9 个可调参数）：**

| 参数 | 类型 | 范围 |
|------|------|------|
| 策略选择 | 下拉（可多选对比） | S-01 ~ S-15 |
| 天气 | 下拉 | 晴天/阴天/小雨/大雨/大雾/大风 |
| 车流量 | 滑块 | 1万~5万 辆/天 |
| 大货车占比 | 滑块 | 10%~60% |
| 分流道路路况 | 下拉 | 畅通/轻度拥堵/中度拥堵/严重拥堵 |
| 港口运力 | 滑块 | 0%~100% |
| 时段 | 下拉 | 早高峰/平峰/晚高峰/夜间 |
| 汇入车流 | 滑块 | 0~500 辆/h |
| 信号灯方案 | 下拉 | 当前方案/优化方案A/优化方案B |

**模拟结果：**
- 多策略预测曲线叠加对比（含"不干预"基线）
- 对比表：缓解时间 / 分流量 / 成功率 / 置信度
- AI 推荐最优组合 + 理由

**数据来源：**
- 初始阶段：基于 STRATEGY_DB 静态参数 + 因素权重公式
- 积累阶段：基于指挥模式复盘数据校准参数

### 6.2 策略进化记录

**用途：** 展示 AI 模型每次迭代变化，让领导看到系统在持续进化

**展示方式：** 时间线 + 详情卡片

每条进化记录包含：
- 版本号 + 日期
- 触发事件（哪次执行/哪条反馈触发的）
- 变化内容（新增/调整了什么因子）
- 影响范围（哪些策略受影响）
- 准确率变化（before → after）

顶部有准确率趋势线，直观展示 AI 在进步。

### 6.3 策略决策树

**用途：** 图形化展示 AI 推荐策略背后的决策逻辑

**交互设计：**
- 交互式树形图，可展开/折叠
- 高亮当前场景匹配的路径
- 每个节点显示历史命中次数和成功率
- 标注哪些分支是 AI 后来学到的（v1.1/v1.2 标签）

### 6.4 策略知识图谱

**用途：** 展示策略之间的关系网络（互斥/联动/因子影响）

**展示方式：** 力导向关系图

- 节点 = 策略（S-01~S-15）+ 影响因子
- 连线 = 联动（蓝色实线）/ 互斥（红色虚线）/ 影响（紫色点线）
- 节点大小 = 使用频率
- hover 显示策略详情，点击高亮关联关系

## 七、总览模式改动

### 7.1 分级触发逻辑完善

从"只看当前值"升级为"预测性分级触发"：

| 等级 | 触发条件 | 系统行为 | 小语播报 |
|------|---------|---------|---------|
| 低风险 | AI 预测 30min 后达 4-5 | 摘要面板变黄 | "预计半小时后车流增大，暂无需处理" |
| 高风险 | AI 预测 30min 后达 >6 | 弹出建议框 | "预计 20 分钟后严重拥堵，建议提前准备" |
| 紧急 | 事故/停航/实际 >6 | 自动切换指挥模式 | "紧急情况，已切换至指挥模式" |

### 7.2 高风险触发时的"预准备"

用户接受"提前进入指挥模式"建议时，系统在切换前已完成：
- 归因预判（基于当前趋势）
- 策略预推荐（提前部署型策略优先）
- 资源预查询（当前可用资源状态）

### 7.3 场景预设按钮（演示用）

隐藏在右上角齿轮图标内：

| 按钮 | 数据状态 | 触发效果 |
|------|---------|---------|
| 日常平稳 | 拥堵指数 2.5 | 小语正常播报 |
| 低风险 | 车流渐增 | 小语播报预警，摘要变黄 |
| 高风险 | 港口积压+车流激增 | 小语紧急播报，建议切换 |
| 紧急事故 | 进港大道事故，指数 7.0 | 自动切换指挥模式 |
| 策略执行中 | S-02 执行 30min | 偏差监控中 |
| 策略完成 | 拥堵缓解，指数 2.9 | 弹出结构化复盘 |

## 八、核心数据结构

### 8.1 策略定义（升级）

```typescript
interface StrategyDefinition {
  // 基本信息
  id: string;
  name: string;
  type: 'diversion' | 'signal' | 'emergency' | 'port' | 'info';
  
  // 推荐理由模板
  reasonTemplate: string;
  triggerCondition: string;
  
  // 所需资源
  requiredResources: {
    type: 'police' | 'cone' | 'led_screen' | 'tow_truck';
    quantity: number;
    estimatedArrivalMin: number; // 历史平均到位时间
  }[];
  
  // 效果模型
  effectModel: {
    baseEffect: number; // 基础效果（拥堵指数降幅）
    factorModifiers: {
      weather_rain: number;    // 雨天修正系数
      weather_fog: number;     // 雾天修正系数
      truck_ratio_high: number; // 大货车>40%修正
      road_congested: number;  // 分流路况拥堵修正
      inflow_high: number;     // 汇入车流>200修正
    };
  };
  
  // 历史数据（由 AI 分析模式反哺）
  historicalData: {
    executionCount: number;
    successRate: number;
    avgReliefMinutes: number;
    conditionMatchRecords: ConditionMatchRecord[];
  };
}
```

### 8.2 执行记录

```typescript
interface ExecutionRecord {
  id: string;
  strategyId: string;
  startTime: number;
  endTime: number;
  
  // 版本迭代
  versions: {
    version: string; // v1, v2, ...
    content: string;
    reason: string;
    expectedCurve: { minutesAfter: number; expected: number }[];
    timestamp: number;
  }[];
  
  // 实际曲线
  actualCurve: { timestamp: number; congestionIndex: number }[];
  
  // 偏差事件
  deviationEvents: {
    timestamp: number;
    type: 'strategy' | 'execution' | 'environment';
    reason: string;
    action: string;
    resolutionMinutes: number;
  }[];
  
  // 资源到位
  resourceArrival: {
    estimated: number; // 预计分钟
    actual: number;    // 实际分钟
  };
  
  // 用户评价
  rating: 'effective' | 'moderate' | 'ineffective';
  comment: string;
  
  // AI 学习
  aiLearnings: {
    newFactor: string;
    affectedStrategies: string[];
    accuracyChange: { before: number; after: number };
  }[];
}
```

### 8.3 进化记录

```typescript
interface EvolutionRecord {
  version: string;     // v1.1, v1.2, ...
  date: string;
  triggerEvent: string; // 哪次执行/反馈触发
  triggerExecutionId: string;
  changeDescription: string;
  affectedStrategies: string[];
  accuracyBefore: number;
  accuracyAfter: number;
}
```
